Flink Jobs

🗺️

Heatmap Aggregator

Maintains a sliding 5-min and full-match window of player XY positions. Normalizes to 20×12 zone grid. Stores as sparse matrix → rendered to SVG tiles on demand. Uses Flink's ProcessWindowFunction with RocksDB state backend. Output: heatmap snapshot every 30s per player.

🔗

Pass Network Builder

Maintains a weighted directed graph per team per match. Nodes are players, edges are pass attempts/successes. Stored in Neo4j. Flink emits graph delta events every time a pass occurs. Downstream: calculates centrality (betweenness, degree), identifies key connectors.

⚽

xG Calculator

Real-time XGBoost inference on shot events. Features: shot location (x,y), angle, distance, body part, assist type, defensive pressure (from tracking), game state, time in match. Model served via TF Serving gRPC. xG accumulates to match total and shown per shot on shot map.

🎯

Shot Map Renderer

Tracks all shot events with coordinates, outcome (goal/save/miss/blocked), xG value, and shooter. Stores in Redis sorted set per match. Computes aggregate on-target %, shots per zone. Rendered client-side as D3 pitch visualization using the shot array from the API.

🏃

Dribble Tracker

CEP (Complex Event Processing) in Flink. Detects dribble pattern: player receives ball → speed vector changes → retains possession past defender within 2m. Labels outcome: successful / unsuccessful. Stores sequence of events, duration, distance covered.

📊

Match Stats Aggregator

Rolling counters per team per match: possession %, shots on/off target, corners, fouls, yellow/red cards, offsides, PPDA, progressive passes. Maintained in Redis HASHes, flushed to ClickHouse every 60s for historical persistence. Served with microsecond cache reads.

Score Reconciliation — Multi-Source Voting

Scores are the most critical data. We ingest from 3+ sources simultaneously and use consensus voting with source-specific trust weights to resolve conflicts. This is the Single Source of Truth for all score-derived data.

class ScoreReconciler { sources: { OPTA: { weight: 0.90, avgDelay: "1.2s" }, SPORTRADAR: { weight: 0.85, avgDelay: "2.1s" }, MANUAL_OP: { weight: 0.75, avgDelay: "4.0s" }, AI_VISION: { weight: 0.60, avgDelay: "8.0s" }, } // Window: accept late events up to 30s (Flink allowedLateness) reconcile(events: GoalEvent[]) → CanonicalScore { if (weightedMajority(events) >= 0.8) publishImmediately() else if (anySourceConfident && noConflict) publishWithFlag("UNCONFIRMED") else holdFor(15s, waitForHumanReview) } // VAR / Review state machine stateTransitions: SCORED → UNDER_REVIEW → (CONFIRMED | DISALLOWED) }

Sport-Specific Design

Football / Soccer

The most data-rich sport. We receive optical tracking at 25fps (50 data points per second per player × 22 players = 1100 coordinate pairs/second per match). This is the highest bandwidth sport.

🟩

Pass Map

Pass origin and destination coordinates stored per player. Client renders a pitch overlay with directional arrows. Line weight = pass frequency. Color intensity = completion rate. Queryable by time window (1st half, full match). Data model: pass_id, from_xy, to_xy, player_id, success, timestamp.

🔥

Player Heatmap

20×12 zone grid (normalized pitch). Flink aggregates position buckets per player in real-time using tumbling windows. Stored as 240-cell float32 array per player per half. Rendered client-side as a gradient overlay using canvas. API serves compressed binary blob ~1KB per player.

⚡

xG & Shot Map

XGBoost model (38 features) produces xG for each shot. Shot map shows all shot positions on pitch with outcome icon and xG value. Match xG timeline shows cumulative xG over time — this is the most requested football visualization. Model retrained weekly on 500K+ historical shots.

💨

Dribble & Touch Maps

Each successful dribble stored with start/end coordinate, defender overcome (player_id), speed delta. Aggregated to show per-player dribble tendency zones and success rate. Powered by Flink CEP detecting possession → speed change → defender bypass pattern in tracking data.

Tennis

Tennis has one of the most elegant statistical visualizations — the shot map. We receive coordinates from Hawk-Eye's ball-tracking system which provides 3D ball position at 60fps.

🎾

Shot Map

Every shot landing position stored with bounce coordinate on normalized court (0–100 × 0–100). Separated by serve direction (T, body, wide), return type (FH/BH), shot type (winner, forced error, unforced error). Rendered as court heatmap and individual dot plot. Win % per zone shown for first/second serve.

📡

Serve Analytics

Serve speed, direction, spin type, ace/fault. Rally length distribution. Second-serve % by direction. Clutch serve stats (break points, tiebreaks). Aggregated per set, per match, per surface. ATP ShotLink API provides coordinates; Hawk-Eye provides 3D ball spin for premium events.

📈

Win Probability

Real-time win probability model updates on every point. Features: current score, set scores, serve advantage, surface, player head-to-head, current momentum (last 5 points won/lost). Shown as oscillating probability gauge — one of the most engaging live visualizations.

Formula 1

F1 is uniquely data-dense — the official F1 Live Timing API streams telemetry including GPS positions, sector times, pit stop data, tyre information, and radio transcripts in near real-time.

🏎️

Race Flow / Position Changes

A Sankey-style visualization showing position changes lap by lap. Each driver is a colored ribbon. Position changes animated on each lap completion. Data stored as lap-by-lap positional matrix (20 drivers × ~70 laps). Kafka topic: f1-timing with 50ms flush. Position changes trigger push notification to fans of specific drivers.

📋

Live Leaderboard

Real-time table showing: Position, Driver, Team, Tyre compound + age, Gap to leader, Last lap time, Best lap, Pit stop count, Sector colors (purple/green/yellow). Updates every 10 seconds from F1 Timing API. Stored in Redis sorted set (score = race position) for O(1) lookups. WebSocket push on every change.

🗺️

Live Track Map

Each car's GPS position plotted on circuit SVG in real-time. DRS zones highlighted. Safety car delta sectors shown. Gap intervals between drivers shown as animated markers. Track map refreshes at 1Hz from the API. Circuit coordinates pre-stored in S3; car positions overlaid via WebSocket.

📻

Team Radio Feed

F1 provides radio clips via its Live Timing API. Audio clips stored in S3 with 5s of latency after broadcast. Metadata: timestamp, driver, clip_url, auto-transcription via Whisper ASR. Shown chronologically in commentary feed. Fan engagement: most replayed clips surfaced in social buzz.

Golf

⛳

Hole-by-Hole Live Scoring

Each hole scored as Eagle/Birdie/Par/Bogey/Double+. PGA ShotLink API provides shot coordinates per hole. Storage: player_id, round, hole_num, par, strokes, shots[{shot_num, from_xy, to_xy, distance, club, lie}]. Leaderboard is cumulative to-par. Real-time leaderboard in Redis sorted set.

🏌️

Shot Tracer

3D shot trajectory stored per shot (arc path as Bezier curve parameters). Course map overlaid with each player's shots for selected hole. Putting line shown on green with final ball position. Proximity-to-pin statistics by tee shot, approach, and chip categories for each round.

📊

Strokes Gained

Strokes Gained (SG) calculated for each shot using baseline PGA Tour model. SG:Driving, SG:Approach, SG:Around-the-Green, SG:Putting computed per round and cumulative. Stored in ClickHouse for historical cross-player comparisons. Key differentiator from traditional +/- scoring.

Social Media Buzz Engine

When a goal is scored or a penalty is missed, social media erupts. The social buzz feature surfaces the most relevant, verified, and engaging reactions and links them to specific match events on a unified timeline. This is a complex NLP + event-linking pipeline running entirely in real-time.

Ingestion → Enrichment → Ranking Pipeline

Reddit · Instagram
→
Keyword Filter

→
NLP Pipeline
entity extraction, sentiment
→
Event Linker

→
Buzz Scorer
→
Social Feed API
🔍

Keyword Filter

Each active match has a dynamic keyword set: team names (all aliases, hashtags, abbreviations), player names + surnames, official hashtags (#MUNCHE, #PL). Maintained in Redis and pushed to Flink. Bloom filter pre-screens the full firehose before heavy NLP runs. ~98% rejection rate at this stage.

🧠

NLP Enrichment

Each candidate tweet/post runs through: (1) Language detection, (2) Named entity recognition (player names, team names), (3) Sentiment classification (positive/negative/neutral), (4) Event type classification (goal celebration, controversy, injury concern), (5) Toxicity filtering — toxic posts excluded.

⏱️

Event Linker

Key insight: social posts spike within 30–120s of a real event. We use temporal correlation: post timestamp vs. match event timestamp. Posts within ±90s of a goal are tagged to that goal event_id. This creates a social activity timeline anchored to match events. Flink session windows of 180s per event.

📈

Buzz Scoring

Each event gets a real-time Buzz Score: (post_volume × recency_decay) + (engagement_weight × verified_bonus). Verified accounts and journalists get 3× multiplier. Score decays with t^-0.7 half-life. Top-5 posts per event curated by engagement. Score powers "Most Talked About" ranking on discover feed.

API Design

The platform exposes four API paradigms to serve different client needs. The choice of paradigm per feature is deliberately matched to the data's update frequency and the client's consumption pattern.

REST API — Static & Historical

Used for: team info, player profiles, league standings, historical match results, tournament brackets. Heavily cached (CDN + Redis). Cache-Control: max-age=60 for standings, max-age=3600 for player profiles.

GET /v3/matches/{matchId}/stats GET /v3/players/{playerId}/heatmap?matchId=&half=1 GET /v3/matches/{matchId}/shots GET /v3/leagues/{leagueId}/standings?season=2024 GET /v3/f1/races/{raceId}/lap-by-lap GET /v3/tennis/matches/{matchId}/shot-map GET /v3/golf/tournaments/{id}/leaderboard

GraphQL — Composite Client Queries

Used for: match overview pages that need score + lineups + stats + social buzz in one request. Federated gateway stitches across microservices. Persisted queries only in production (no arbitrary query injection).

query MatchOverview($matchId: ID!) { match(id: $matchId) { score { home away } stats { possession xg shots } heatmaps { player { id name zones } } shotMap { shots { x y xg outcome } } socialBuzz { topPosts { content buzz } } } }

WebSocket — Live Updates

Clients subscribe to a match room on connect. Server pushes delta events only (not full state). Client maintains local state and merges deltas. Each WebSocket node handles 1M concurrent connections using Netty/io.vertx. Nodes are stateless — subscription state stored in Redis.

// Client → Server { "action": "subscribe", "matchIds": ["m1","m2"] } // Server → Client (delta push) { "type": "GOAL", "matchId":"m1", "minute":67, "team":"home", "scorer":"Salah", "score":{"h":2,"a":1}, "xg_total":{"h":2.41,"a":0.87} } { "type": "SHOT", "matchId":"m1", "x":78, "y":42, "xg":0.22, "outcome":"save" } { "type": "STAT_UPDATE", "matchId":"m1", "possession_home":58, "shots_home":12 }

SSE — Commentary Feed

Server-Sent Events for the commentary feed. Unidirectional — server pushes commentary lines as they arrive. Automatically reconnects. Falls back to long-polling for environments that don't support SSE. Sequence IDs allow resumption after reconnect without missed events.

Unique Constraints & Solutions

⚡ The World Cup Final Spike — 50× Normal Traffic in 2 Seconds

A Mbappe goal in the 90th minute of a World Cup final will trigger 50M simultaneous push notifications, 50M WebSocket messages, and 200M HTTP requests in the following 60 seconds. This is a 50× traffic spike with essentially zero warning time — the event itself is the trigger.

→ Solution

Pre-warmed auto-scaling: For matches with >1M tracked fans, we pre-scale WS gateway fleet 30 minutes before kickoff. Push notification fan-out uses topic-based FCM/APNs so a single server-side publish reaches millions — we never loop over individual tokens at the application layer. Redis pub/sub is horizontal: 1 publish → N gateway nodes each deliver to their local subscribers in parallel. WS node fleet scales to 1000 nodes using k8s HPA on connection-count metric. CDN serves all static assets eliminating 95% of origin HTTP load.

⚖️ Multi-Source Score Conflict — Three Sources Say Different Scores

During a controversial VAR review, Opta may say goal, Sportradar may not have processed it yet, and our manual operator is also uncertain. Publishing the wrong score destroys user trust. This is a consistency vs. latency tradeoff that must be tuned per event severity.

→ Solution

The Score Reconciler implements a weighted consensus protocol. For goals (highest-stakes events), we require 2-of-3 sources to agree before publishing as CONFIRMED. We immediately publish an UNCONFIRMED goal with an "awaiting confirmation" badge — users see it fast, but it's flagged. If sources disagree for >30s, a human operator alert is triggered (PagerDuty → on-call manual review). The match enters SCORE_REVIEW state. VAR states are tracked explicitly in the state machine: SCORED → UNDER_VAR → CONFIRMED / DISALLOWED.

🕐 Tracking Data at 25fps — 1100 Coordinates Per Second Per Match

Optical tracking at 25fps for 22 players = 550 positions/second per match. With 100 simultaneous matches using tracking, that's 55,000 coordinate pairs/second ingested. This is purely for internal processing — clients cannot receive raw tracking data. We must aggregate intelligently.

→ Solution

Raw tracking data is written to TimescaleDB (time-series optimized) and simultaneously streamed to Flink for real-time aggregation. Flink maintains in-memory rolling aggregates (heatmap zone counters, player distance, speed stats) using RocksDB state backend. Clients never receive raw tracking — only derived artifacts: heatmap zone array (240 bytes), distance covered (8 bytes per player), top speed (4 bytes). Heatmap images are pre-rendered server-side into PNG tiles every 60s and stored in S3, served via CDN. Zero tracking data crosses the API boundary.

📡 Network Partitions During Live Matches — Data Source Goes Down

The Opta feed goes down mid-match. Our ingestion agent loses connection. We cannot drop updates — a 5-minute gap in the score/event stream would be catastrophic for user experience.

→ Solution

Every data source has a hot backup source configured in a priority chain. If Opta fails, Sportradar becomes primary within 10s (circuit breaker trips after 5 failures). Manual operators are alerted immediately and begin supplementing critical events (goals, cards) manually via the internal admin tool. The commentary feed continues via the backup source. Flink jobs use Kafka's replay capability: if a source reconnects and has buffered events, they're replayed in sequence and late events are accepted within a 30s allowed lateness window. Clients see a "Live data delay" indicator if our P99 freshness exceeds 10s.

🌍 Global Latency — Users in Southeast Asia Watching European Football

A fan in Jakarta watching a Premier League match in London. The event happens at 19:00 UTC in London. Our nearest data center is in Frankfurt. Jakarta is 7000km from Frankfurt. Without optimization, round-trip latency for WebSocket push is 150ms+ just for network propagation.

→ Solution

Active-active multi-region deployment in 6 regions: US-East, EU-West, SE-Asia, India, South America, Oceania. Kafka MirrorMaker2 replicates events cross-region with ~500ms lag acceptable (total fan delivery: 1500ms including regional processing). WebSocket gateway in Singapore serves Jakarta users — network hop is 30ms instead of 150ms. Redis is replicated read-only to all regions; score writes always go to EU-West (primary for EPL) then async-replicated. For live events, the primary-region latency advantage (~1–2s) is imperceptible to fans.

🏅 Olympics — 10,000 Athletes, 40 Sports, 336 Events in 17 Days

The Olympics is a unique operational challenge: extreme burst of new sports coverage, new athlete profiles, unusual event formats (heats/semis/finals with complex qualification logic), and 17 days of maximum global attention.

→ Solution

Olympics has a dedicated ingestion pipeline using the IOC's ODF (Olympic Data Feed) XML format over SFTP. A dedicated Olympics Service handles the sport-specific normalization for 40 different event types. Athlete profiles and event schedules are pre-loaded 2 weeks before Games start. Dynamic sport templates are configured via a sport config JSON stored in S3, allowing new sports to be added without code deploys. Stadium-specific timezone handling + broadcast schedule integration. Operational scale-up: 2× WS gateway fleet pre-deployed from Opening Ceremony. 48-hour on-call rotation for engineering team during Games.

🤖 AI xG Model Staleness — Model Gives Outdated Probabilities

The xG model is trained on historical data. In a match played on artificial turf in heavy rain, the model (trained on dry natural grass) may produce systematically biased xG values. Serving a stale model silently is worse than no model.

→ Solution

xG model is retrained weekly using the past 2 seasons of data, incorporating surface type, weather, and competitive context as features. Model versions are managed in MLflow. A/B testing framework compares new model xG predictions vs. actual goals on held-out matches — if Brier score degrades >0.05, rollback is automatic. Model confidence score is included in every xG output. If confidence <0.5 (e.g., unusual match context), the UI shows "xG unavailable" rather than a potentially misleading number. Feature store is maintained in Redis for sub-5ms feature lookup during inference.

Caching Strategy — The Read Amplification Problem

With 50M concurrent users watching the same match, every cache miss is 50M database hits. We use a multi-tier caching strategy with aggressive TTLs and targeted cache invalidation:

🥇

L1 — CDN Edge Cache

All static assets (heatmap PNGs, player photos, court diagrams) cached at Cloudflare edge in 300+ PoPs. TTL: 60s for live stats tiles, 7 days for images. This absorbs >95% of all reads globally before they reach origin.

🥈

L2 — Redis Application Cache

Live match state, leaderboards, F1 positions. Sub-millisecond reads. TTL: 0 (persistent until invalidated by event). Invalidation on every score/stat update via Kafka-triggered CDC. 64-shard cluster, 3.2TB RAM.

🥉

L3 — Read Replicas

PostgreSQL + TimescaleDB read replicas per region. Handles historical data queries (player profiles, past match stats). Cache-aside pattern: read from replica, cache result in Redis for 5 min TTL on rarely-changing data.

CDN & Edge Strategy

🌐

Cloudflare Workers — Edge Logic

Lightweight JS functions at the edge handle: authentication token validation, geo-based sport preferences (cricket popular in India, NHL in Canada), A/B test bucket assignment, and regional content filtering. Executed in 300+ PoPs in <5ms — no origin hit needed for auth checks.

📦

Static Asset Strategy

Heatmap PNG tiles (pre-rendered every 60s per player), court/pitch SVG templates, team logos, player photos — all stored in S3 with Cloudflare CDN in front. Cache-Control: public, max-age=60 for live tiles. Immutable for player/team images (keyed by hash). 95%+ cache hit rate.

⚡

WebSocket Routing

Cloudflare Durable Objects maintain sticky WebSocket sessions at edge. Each WS connection is proxied through the nearest PoP, reducing RTT. Durable objects ensure WS session state survives origin node restart. Anycast routing connects users to their nearest WS gateway region.

📱

Mobile Optimization

Native apps use Brotli-compressed protobuf responses instead of JSON (60% smaller payloads). Heatmap data is sent as sparse array (only non-zero zones). WebP player photos served via CDN. On poor connectivity, SSE falls back to adaptive polling (5s → 10s → 30s) to preserve battery.

Reliability & Operational Excellence

Achieving 99.99% Uptime

Redundancy

Every component is deployed with N+1 redundancy minimum. Kafka: 3-broker clusters with RF=3. Redis: master + 2 replicas per shard. PostgreSQL: primary + 2 synchronous standbys. All services deployed across 3 AZs. Multi-region active-active for read path; active-passive for write path with <30s RTO.

Circuit Breakers

Every service-to-service call and external API call is wrapped in a circuit breaker. If a downstream service is degraded, upstream services fail fast and use cached/fallback data. This prevents cascading failures where a slow third-party data provider takes down the entire ingestion pipeline.

Graceful Degradation

If real-time tracking fails: show last-known heatmap. If xG model fails: hide xG, show raw shots only. If social buzz fails: hide the section. If websocket fails: fall back to 5s polling. Clients are designed to handle partial data — every section independently loads. No single API failure takes down the entire match page.

Chaos Engineering

Weekly chaos experiments using Chaos Monkey and custom Flink failure injection. We regularly kill random WS gateway nodes, inject Kafka consumer lag, and simulate data source failures during off-peak hours. Runbooks are auto-generated and tested quarterly. On-call rotation with 5-minute response SLA for P0 (score wrong) incidents.

Observability Stack

📊

Metrics

Prometheus + Grafana. Key dashboards: event ingestion lag per source, score update P99 latency, WebSocket fan-out time, cache hit rates per endpoint, Kafka consumer group lag, Flink checkpoint times, xG model inference latency.

🔍

Distributed Tracing

Jaeger for end-to-end tracing of events from ingestion to client push. Every event carries a trace_id through Kafka headers. Critical path: "time from source event to client WebSocket push" — target P99 <500ms, alerting on P95 >300ms.

📝

Logging

Structured JSON logs via Filebeat → Elasticsearch → Kibana. Every score update logged with source, timestamp, match_id. Score discrepancies automatically create an incident ticket. Log retention: 30 days hot, 365 days cold (S3 Glacier).

🚨

Alerting

PagerDuty integration. P0 alerts: wrong score published, ingestion lag >60s for live match, WS gateway <50% capacity. P1 alerts: xG model down, social buzz pipeline down, tracking data gap >30s. All resolved within SLA or auto-escalated to engineering lead.

🏆 Key Design Philosophy: The system is designed around the principle that scores must never be wrong. Every other feature (heatmaps, social buzz, xG) can degrade gracefully — but a wrong scoreline destroys user trust irreparably. The Score Reconciler, multi-source consensus, and human operator fallback all exist to protect this invariant above all others.
