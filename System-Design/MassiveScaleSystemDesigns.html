Detailed Design

Order Checkout Saga: Saga orchestrator (Step Functions or custom) coordinates: (1) Lock cart via Redis SETNX, (2) Reserve inventory via Inventory Service (optimistic decrement with rollback), (3) Create payment intent via Stripe/internal PSP, (4) On payment success → publish OrderConfirmed to Kafka → emit to Fulfillment/Notification. On any failure → compensating transactions release inventory, void payment intent, unlock cart.

Inventory Management: Two-tier: Redis counter for hot-path real-time deductions (DECR atomic), Postgres as source of truth sync'd every 30s via background job. Flash sales use Lua scripts in Redis: DECR only if counter > 0 in a single atomic operation, preventing oversell even at 50K req/s.

Search Architecture: Elasticsearch with 5 shards/2 replicas per index. Query-time function scoring: relevance score × boost(rating) × boost(sales_rank) × seller_quality. Delta sync from Kafka events (product updated → re-index within 5s). "Suggest" index (smaller, prefix-optimized) for autocomplete via edge n-grams.

Caching Strategy: L1 = in-process LRU (Caffeine, 10K entries, 10s TTL); L2 = Redis cluster (product pages 5min TTL, category pages 1min, search results 30s). Cache invalidation via Kafka consumer on ProductUpdated events. Write-through for product data, write-behind for view counts.

Strong Consistency: Inventory

Redis DECR + Postgres 2PC for inventory prevents oversell; justified for revenue impact.

Eventual Consistency: Catalog

Price updates may be 5s stale in search; acceptable — checkout always re-validates live price.

CQRS for Orders

Write model: normalized Postgres. Read model: DynamoDB + S3 projections for order history queries.

Cart Consistency Trade-off

DynamoDB eventual consistency means concurrent multi-device cart edits may conflict. Last-write-wins with client-merge UX.

⚠ Interviewer Traps

  • Don't say "just use a DB transaction for checkout" — at 10K TPS, distributed saga is mandatory; explain compensating transactions explicitly
  • Oversell prevention: naive SELECT FOR UPDATE won't scale; Redis DECR or optimistic locking with version field is the answer
  • Search: don't use DB LIKE queries; Elasticsearch with inverted index, scoring functions, and delta sync is expected
  • Cart: DynamoDB not a relational join; denormalize and store price snapshots at add-time, re-validate at checkout
  • Reviews: async pipeline required — sync write then async ML moderation/aggregation
07

Distinctive Component: Flash Sale Engine

⚡ Sub-10ms Inventory Deduction at 100K TPS

Amazon's "Lightning Deal" problem: 1M users simultaneously hit "Buy Now" for 500 units. Solution architecture:

  • Pre-warm: 5 min before sale, load inventory count into Redis as a counter per SKU
  • Lua Script (atomic): if redis.call('GET',key) > 0 then DECR; return 1 else return 0 end — single round trip, no race condition
  • Queue overflow: Excess requestors enter a virtual waiting room (SSE-based), shown position in queue
  • Async Postgres sync: Every 10s, Redis counter reconciled with Postgres inventory truth via background job with idempotent upsert
  • Rollback on payment failure: If Stripe declines, INCR the Redis counter and publish InventoryReleased event
  • Anti-bot: CAPTCHA + rate-limit per user_id (Redis sliding window 60s/10 attempts) at API Gateway

Detailed Design

Bid Acceptance — Lua Script in Redis: The entire bid validation is a single Lua script: read current_price, check bid > current_price + increment, check auction not expired, check bidder not current winner, THEN atomically set new price and winner. Returns 0/1. This makes bid acceptance lock-free at the Redis level. Postgres write happens async (best-effort durable log) via Kafka consumer — if it fails, retry with idempotency.

Proxy Bidding: When bid service processes a new bid, it checks all existing proxy bids for this auction (stored encrypted in Redis sorted set by proxy_max). If current bid triggers a proxy, automatically generate counter-bids up to proxy_max. This happens inside the same Lua transaction to maintain atomicity.

Anti-Sniping — Auction Extension: Scheduler uses Redis sorted set with auction_id scored by end_time. A background poller (every 100ms) checks for auctions ending within 2 min. When a bid arrives in the last 2 min, Auction Service does: ZADD auctions_schedule {new_endtime} {auction_id} atomically with bid commit. Broadcast extension event to all watchers.

WebSocket Scalability: 1,000 WS servers, each holding 50K connections. When Kafka consumer (fanout service) receives BidPlaced event, it looks up Redis Set watcher:{auction_id} → gets list of WS server IDs → uses Redis Pub/Sub to push to those specific servers → servers push to individual connections. Consistent hashing maps auctions to WS server partitions.

⚠ Interviewer Traps

  • Don't accept bids with just a DB transaction — Postgres row lock at 50K TPS will deadlock; Redis Lua is the answer
  • Auction end time: don't rely on application server clocks — use Redis TTL or NTP-synced scheduler with at-least-once semantics + idempotent close handler
  • Proxy bid: must be server-side and encrypted at rest — never expose proxy max to other bidders or it's fraud enablement
  • WebSocket at scale: can't broadcast to 50M connections from one service — partition by auction, use Redis Pub/Sub for server-to-server coordination
  • Winning bid confirmation: "winning" status in Redis is provisional; only Postgres write + payment success = final win
07

Distinctive Component: Real-Time Bid Consistency

🔒 Exactly-Once Winning Bid Guarantee

The hardest problem: two bidders simultaneously place the exact same bid amount. Solution uses a 3-phase commit with Redis as coordinator:

  • Phase 1 — Redis CAS: Lua script does GETSET + version check atomically. Only one thread wins
  • Phase 2 — Async Postgres: Winner publishes BidAccepted with bid_id to Kafka. Consumer writes to Postgres with ON CONFLICT DO NOTHING on (auction_id, placed_at, amount)
  • Phase 3 — Reconciliation: Every 60s, a reconciler compares Redis state to Postgres and flags discrepancies for manual review (SLA breach alerting)
  • Shill bidding detection: Graph analysis of bidder–seller relationships; bidders who only ever bid on the same seller's auctions, bid amounts always just above previous, device fingerprint clustering → auto-flag + human review queue

Detailed Design

Recurring Event Expansion: Never store all occurrences — store RRULE string. At query time, Calendar Service uses iCal4j (Java) or python-recurring-ical-events library to expand RRULE in the requested [timeMin, timeMax] window. Exception instances (edited single occurrences) stored as separate event rows with original_id FK pointing to the master; merged during expansion. Deletion of single instance = cancelled exception record.

Free/Busy with Redis BITFIELD: 24 hours × 4 slots/hour = 96 bits per user per day. On event create, Calendar Service atomically sets the relevant bits. FreeBusy Service reads Redis BITFIELDs for all requested users and ORs them together — sub-millisecond for 20 attendees. Near-term 7 days in Redis; historical falls back to DB aggregation query.

Delta Sync Token: Sync token = opaque encoding of (user_id, last_sync_timestamp). On each sync request, query WHERE updated_at > last_sync_time AND calendar_id IN (user_calendars). Return changed/deleted events + new token. Token expired (>30 days) → 410 GONE → client does full sync. Mobile clients batch sync every 15 min; push channels trigger immediate sync on change.

Reminder Delivery — DynamoDB as a job store: Partition key = fire_minute (epoch minute). A fleet of reminder workers query DynamoDB for the current minute's bucket every 60s, send notifications (FCM for Android, APNS for iOS, SES for email), mark sent=true. Idempotent by reminder_id — at-least-once delivery with dedup.

⚠ Interviewer Traps

  • Never materialize all recurrence instances — "weekly for 3 years" = 156 rows is manageable, but "daily for 5 years" = 1825 rows × 1B users breaks storage; always expand lazily at query time
  • Timezone correctness: store all times as UTC; convert to user's timezone only at read time; DST transitions must re-evaluate recurrence on every query
  • Editing recurring events: "this only", "this and following", and "all" are three distinct write patterns; "this and following" splits the series into two master records
  • Free/busy privacy: when user sets visibility=private, return BUSY intervals but not event details to non-owners
  • Sharding: shard by user_id not calendar_id — calendar queries always join to a single user's data

Distinctive Component: Smart Scheduling + Find-A-Time

🧠 Sub-100ms Meeting Slot Finder for 50 Attendees

  • Input: 50 attendee emails, desired duration 60 min, within next 7 days, 9am–6pm
  • Step 1: Batch Redis BITFIELD reads for all 50 users × 7 days = 350 BITFIELD GETs (pipelined, ~2ms)
  • Step 2: Bitwise OR all 350 bitmaps (96-bit bitwise union) → find zero-runs ≥ 4 consecutive slots (60 min)
  • Step 3: Rank candidates by: fewest conflicts, business hours weight, attendee timezone spread, preferred meeting times from ML model
  • Step 4: For users missing from Redis cache, async DB fallback; return partial results with confidence score
  • Working hours: Stored per user as a weekly RRULE in Redis; applied as mask before slot search
  • Total latency: ~20ms Redis + 5ms compute + 10ms cache miss fallback = well under 100ms

Detailed Design

Audio Streaming: Tracks pre-encoded at 3 bitrates + lossless. Each track split into 10-second segments (HLS/DASH). Stream Service generates pre-signed S3/CDN URLs valid 5 min. Client prefetches next 3 segments ahead. CDN edge caches top 5% of tracks (Pareto — covers 85%+ of streams). Long-tail tracks fetched from S3 origin. Adaptive bitrate: client switches quality based on measured bandwidth.

Top K — Real-time Leaderboard: Kafka topic partitioned by track_id. Apache Flink job: 30-second tumbling window, counts plays per track per country. ZINCRBY to Redis ZSETs. For global top 50: ZREVRANGE O(K log N). Approximate Top-K using Count-Min Sketch in Flink for memory efficiency (actual ranking confirmed against exact counters every 5 min). Track of leaderboard changes pushed via SSE to subscribed web clients.

Discover Weekly — ML Pipeline: Input: 600M users × listen history (last 6 months). Spark ALS collaborative filtering: user-song interaction matrix factorized into 200-dim latent vectors. For each user: find K nearest neighbor users (cosine sim in vector space) → take their most-played songs you haven't heard → filter by audio feature compatibility (same energy/tempo range as your top songs) → rank by predicted play probability → select top 30. Generated in 4-6 hours on Monday morning on a cluster of 500 Spark executors. Output written to DynamoDB partitioned by user_id.

Cross-device handoff: Player state (current track, position_ms, is_playing, device_id, queue) stored in Redis keyed by user_id with 1s TTL refresh. When user switches device, new device polls player state and resumes from stored position_ms. WebSocket connection per user for real-time device list and state sync.

  • Discover Weekly: don't say "collaborative filtering on every request" — it's batch-computed weekly; storing results in DynamoDB is the key insight
  • Audio streaming: don't stream from your app servers — signed CDN URLs. The app server only handles auth + metadata, audio bytes flow CDN → client
  • Play counting for royalties: needs exactly-once semantics eventually. Kafka + idempotent Cassandra writes + batch reconciliation to rights management system
  • Offline mode: encrypted segments stored on device, DRM key validated against server on reconnect, key revoked if subscription lapses

Detailed Design

Tiered Crawl Scheduling: Products classified into Hot (popularity > 1000 watchers → every 15min), Warm (10-1000 watchers → every 6hr), Cold (<10 watchers → once/day). Redis ZSET scored by next_crawl_at (Unix timestamp). Scheduler runs every second: ZPOPMIN N up to batch_size → push to Kafka → after crawl completes, re-schedule: ZADD {next_crawl_time} {product_id}. Tier re-classification runs nightly based on watcher count changes.

Deduplication on Ingest: Parser extracts price. Before writing to Cassandra, compare to Redis cached current price. If price unchanged → skip write (saves ~80% of Cassandra writes). If changed → write to Cassandra + update Redis cache + publish to price-changes Kafka topic. price_changes topic triggers alert matcher and stats updater.

Alert Matching with Bloom Filter: Product-level Bloom filter in Redis answers "does any alert exist for this product?" in O(1). False positive rate ~1% (check DB unnecessarily 1% of time). True negative = zero DB reads. On Bloom hit: query Postgres for active alerts WHERE product_id=? AND target_price >= new_price → for each match, enqueue notification job to SQS. Alert deduplication: alert triggers once, then becomes inactive (or re-arms after price rises above target + hysteresis band).

Distributed Crawling at Scale: 380 crawler servers, each running 100 async Playwright/Puppeteer or httpx workers. IP rotation: residential proxy pool (rotating every 10 requests per domain). Per-domain rate limiting: Redis sliding window counter per domain per IP. Crawl failure → exponential backoff, re-enqueue to Kafka with retry count. Anti-bot detection: randomize user-agent, request timing, viewport; headless browser for JS-heavy pages. robots.txt fetched and cached per domain, respected.

Cassandra for Time-Series

TWCS compaction perfectly aligns with append-only price history. Wide partition (product+retailer) → one disk seek for full history.

Dedup at Ingest vs Storage

Only storing price-change events (not every crawl) reduces Cassandra writes by 80% but means chart queries must know the "last known price" at any point-in-time.

Crawl Legality Trade-off

Scraping violates many retailers' ToS. Production systems use official APIs where available (Amazon PA API), scraping only where no API exists. Anti-bot measures escalate: CAPTCHAs may require human solving or ML.

Pre-aggregated Stats

All-time low, year low, 30d avg pre-computed by Stats Updater on each price change → O(1) reads vs expensive time-series queries.

⚠ Interviewer Traps

  • Don't put price history in PostgreSQL with a B-tree index — at 30 TB and 500M writes/day, Cassandra's time-series model (TWCS, wide partitions) is the right answer
  • Naive alert matching = SELECT * FROM alerts WHERE target_price >= new_price for every crawled product at 76K/sec = DB meltdown. Bloom filter is the key optimization
  • Crawling 2.5B product-URLs: can't re-crawl everything equally. Tiered scheduling by popularity is how you stay within rate limits while giving good coverage
