6. Detailed Design▼

Slot Calculation Algorithm

For a given date + event_type: (1) Expand recurring availability rules for the date. (2) Subtract calendar busy blocks (fetched from Google/Outlook via cached sync token). (3) Subtract existing bookings from DB. (4) Apply buffer_before/after to remaining gaps. (5) Chunk into duration-sized slots. This runs in ~5ms per day from cache; Redis stores the resulting slot bitmap per user per date (TTL 5 min).

Double-Booking Prevention

Booking creation uses a distributed lock: Redis SETNX on key lock:user:{host_id}:{slot_start} with 10s TTL. If lock acquired: insert booking + add to calendar, release lock. On failure (lock held): return 409. Additionally, DB has a UNIQUE constraint on (host_user_id, start_time) as a safety net. Lock ensures only one inflight write at a time.

Calendar Sync Strategy

Full sync on connect; incremental sync via Google Calendar sync tokens (returns only changed events). Webhook push from Google/Outlook triggers cache invalidation for affected user+date. Polling fallback every 15 min for providers without webhooks. Calendar busy blocks stored in Redis sorted set by start_time for fast range queries.

Timezone Handling

All times stored in UTC. Availability rules stored as day-of-week + local time + user's IANA timezone. On slot calculation, rules are expanded into UTC windows for the target date (handling DST transitions). Invitee timezone is captured at booking time for display and reminder formatting only.

Design Choices

  • ✓ Cache slot lists aggressively (5min TTL) — reads >> writes
  • ✓ Event sourcing for booking state changes (audit trail)
  • ✓ Shard by host_user_id — all a user's bookings co-located
  • ✓ Idempotency key prevents duplicate bookings on retry

Tradeoffs

  • ⚠ Slot cache may serve stale availability (5min window)
  • ⚠ Calendar sync failures mean stale busy blocks
  • ⚠ Redis lock adds latency on booking creation
  • ⚠ Cross-shard queries for team scheduling are expensive
7. Distinctive Component: Conflict-Free Slot Engine + Distributed Booking Lock▼

The hardest part of Calendly is computing available slots correctly across multiple real calendars in real time and preventing double-booking under race conditions. The slot engine must handle: recurring rules, one-off overrides, cross-calendar busy blocks, booking buffers, and DST-aware timezone math — all sub-200ms.

6. Detailed Design▼

Geo-Spatial Search

Elasticsearch with geo_point fields indexed on listing lat/lng. Search query: geo_distance filter + availability filter + price range + amenity terms. ML ranker re-scores top 200 hits using user history, review score, superhostier status, price competitiveness. Result: ranked list of listing cards. Map view: viewport-bounded query with geo_bounding_box. Search index synced from PostgreSQL via CDC (Debezium + Kafka).

Double-Booking Prevention

Availability stored in availability_calendar table as per-date rows. Booking flow: (1) Optimistic lock on rows for the date range. (2) Check all dates = 'available'. (3) Update to 'booked'. (4) Insert booking row. All in a single DB transaction with SERIALIZABLE isolation. If concurrent request wins, second gets serialization failure and returns 409. Redis cache invalidated on success.

Payment Escrow

Payment captured at booking but held in Stripe Connect escrow. Released to host 24h after guest check-in. Cancellations trigger refunds per policy. Service fee retained immediately. Webhooks from Stripe update booking status asynchronously. Saga pattern: if post-payment booking write fails, compensating transaction refunds guest.

Photo Pipeline

Host uploads to pre-signed S3 URL. S3 event → Lambda → resize to 5 standard sizes (thumbnail, small, medium, large, original) → store in S3 → CDN cache. Average processing: 3s. Failed processing retried via SQS DLQ. Perceptual hash checked to flag duplicate/inappropriate uploads.

Design Choices

  • ✓ Separate search (ES) from source of truth (PostgreSQL)
  • ✓ Per-date availability rows: simple, cheap to query/lock
  • ✓ Saga for payment → booking atomicity
  • ✓ Two-sided review reveal prevents retaliatory reviews

Tradeoffs

  • ⚠ ES index lag (eventual consistency with PG)
  • ⚠ Per-date rows → large table for long-range listings
  • ⚠ Serializable transactions → lower booking throughput
  • ⚠ Address masking means GPS-based guests need post-confirm unlock
7. Distinctive Component: Geo-Spatial Search + Payment Escrow + Trust & Safety▼

Airbnb's unique complexity is the triangulation of: (1) geo-spatial ranked search at scale, (2) transactional availability locking across distributed nodes, and (3) a two-sided payment escrow with fraud detection. These three systems interact every time a booking is made.

Inventory Management (Core Problem)

Each inventory row (room_type + date + rate_plan) stores available_rooms as an integer. Booking decrements using: UPDATE inventory SET available_rooms = available_rooms - N WHERE available_rooms >= N AND room_type_id = ? AND date BETWEEN ? AND ? — atomic check-and-decrement. If 0 rows updated, booking fails. Channel Manager API upserts hotel-pushed availability changes, triggers Redis cache invalidation and Kafka event for ES index update.

15-Minute Reservation Hold

On "Reserve" (before payment): create a reservations row with TTL=15min, decrement available_rooms. Background job expires stale reservations and restores inventory. This prevents showing rooms as available when another guest is completing payment. Redis stores hold metadata for fast TTL management.

Design Choices

  • ✓ Atomic UPDATE (not SELECT+UPDATE) prevents race on room count
  • ✓ Rate snapshot in booking row — price changes don't affect past bookings
  • ✓ Cancellation policy snapshotted at booking time
  • ✓ Separation of inventory (write-heavy) from search (read-heavy)

Tradeoffs

  • ⚠ Per-row locking under high concurrent demand (popular hotels)
  • ⚠ 15-min hold → users may see "last room" incorrectly locked
  • ⚠ Channel manager sync failures → stale inventory → potential overbooking
  • ⚠ Multi-room booking across dates requires transaction over many rows

Booking.com must propagate changes to its search index and inventory cache within seconds.

Cache-First Fan-Out Strategy

On search: (1) Check Redis for matching route+date key. Cache hit (20min TTL) → return immediately. Cache miss → (2) Fan-out in parallel to all suppliers covering that route (from supplier_configs coverage map). (3) Responses stream back; aggregator merges, deduplicates, and ranks by price. (4) First supplier response returned to client within 1s. Full results in ~3-5s. (5) Best result set written to Redis cache.

Supplier Rate Limiting

Each supplier connector has a token bucket enforcing max_qps. Requests exceeding quota are queued or dropped (with cache serving stale data). Popular routes pre-warmed via background jobs. GDS suppliers (Amadeus, Sabre) are expensive — their responses are cached longer (1hr) since fares change less frequently than LCC direct APIs.

Price Alert Engine

Alert worker runs every 30min: reads all active alerts from DB, groups by route, checks fare_cache for current cheapest price. If current price drops below last_seen_price by >5% (or below target_price): publish to notification queue → email/push. Update last_seen_price. For "watch this route" alerts with no target: alert on any significant drop (>10%).

7. Distinctive Component: Async Fan-Out Meta-Search + Fare Cache▼

Skyscanner never holds inventory — it's a price aggregation engine. The unique challenge is querying 100+ heterogeneous supplier APIs simultaneously, handling partial failures gracefully, merging results into a coherent ranked list, and caching results while respecting that fares are volatile. The cache-first strategy is the performance backbone.

Live State Management

Redis stores current state for each aircraft: HSET flight:{icao24} lat lng alt speed heading. GEOADD flights_geo {lng} {lat} {icao24} for viewport queries. Flink consumer updates Redis on every position message. Aircraft not updated in 30s are marked stale (TTL-expired or explicit delete). Redis Pub/Sub broadcasts updates to WebSocket servers by geographic shard.

Multi-Source Deduplication (MLAT Fusion)

The same aircraft may be seen by 5-20 receivers simultaneously. Flink deduplication: group by icao24 in 5s tumbling window, pick message with strongest signal (highest msg_count from source). For MLAT (Multilateration, for non-ADS-B aircraft), time-difference-of-arrival from 4+ receivers is fused to compute position. Result: one canonical position per aircraft per 5s interval.

Viewport Diff Streaming

WebSocket server maintains per-connection viewport bounds. On each global update cycle (5s): compute aircraft within viewport, diff against last sent state, push only changes (new aircraft, moved aircraft, departed aircraft). Delta compression reduces bandwidth by ~80%. H3 hexagonal spatial index partitions the globe into cells; WebSocket servers subscribe to Kafka partitions by H3 cell.

6. Detailed Design▼

Activity Upload Pipeline

Client uploads FIT/GPX to S3 pre-signed URL directly (bypasses API servers). S3 event triggers SQS → Activity Processor worker: (1) Parse FIT/GPX → extract GPS stream, HR, power, cadence arrays. (2) Apply privacy zone masking — clip GPS polyline within configured radius of home address. (3) Encode summary polyline (Google Encoded Polyline Algorithm). (4) Compute summary stats (distance, elevation, pace zones). (5) Write to activities table. (6) Publish activity_uploaded Kafka event. Typical processing time: 5–15s for a 1h ride.

Segment Matching Algorithm

The core algorithmic challenge. On activity_uploaded: Segment Matcher fetches all segments whose bounding box overlaps the activity's bounding box (spatial index query). For each candidate segment, runs a map-matching algorithm: find the subsequence of GPS points that best fits the segment polyline within a tolerance (Fréchet distance or hidden Markov model). If matched: compute elapsed time between entry/exit GPS timestamps → insert segment_effort. Update segment leaderboard in Redis sorted set. Trigger PR check vs athlete's previous best. Trigger KOM check with Redis atomic compare.

Social Feed: Hybrid Fanout

Feed is the most read-heavy path. Two strategies blended: (1) Fanout-on-write for normal athletes (<10K followers): on activity upload, worker reads follower list, pushes activity_id to each follower's Redis feed list (LPUSH, capped at 500 items). (2) Fanout-on-read for high-follower athletes (pro cyclists, popular creators): activity_id stored in a separate "celebrity" list; on feed read, merge celebrity list with personal feed list. Feed read: fetch activity_ids from Redis list → batch fetch activity metadata from DB/cache → return ranked feed. Morning spike handled by pre-warming feed caches during low-traffic window.

Segment Leaderboards

Overall KOM/QOM leaderboard: Redis sorted set per segment_id, scored by elapsed_time_s (lower = better, so store as negative or use ZRANGEBYSCORE with inverted range). On new effort: ZADD segment:{id}:overall {time} {athlete_id}. Rank lookup: ZRANK O(log N). Filtered leaderboards (age group, weight, following) too numerous to pre-cache — computed on read from segment_efforts table with appropriate filters. Local Legend: athlete with most efforts in rolling 90 days — computed daily via batch job from segment_efforts.

Global Heatmap

Privacy-critical feature. Cannot use individual activity GPS traces directly. Process: Weekly Spark batch job reads all public activity streams from S3. Applies rasterization: GPS points binned to H3 hexagonal grid cells at multiple zoom levels. Cell density normalized and rendered to PNG tiles at each zoom. Tiles stored in S3, served via CloudFront. Private activities, privacy-zone-masked segments, and activities in sensitive areas (military bases flagged by polygon database) excluded from heatmap entirely.

Design Choices

  • ✓ S3 direct upload — API servers not in write path for large files
  • ✓ Async segment matching — upload instant, enrichment follows
  • ✓ Hybrid feed fanout — avoids celebrity problem on write
  • ✓ Encoded polyline for summary — 10× smaller than raw lat/lng array
  • ✓ Privacy zone masking server-side — client never uploads masked coords

Tradeoffs

  • ⚠ Segment matching is CPU-expensive — scales poorly for huge segments
  • ⚠ Fanout-on-write: if athlete gains followers quickly, fanout list stales
  • ⚠ KOM detection requires atomic compare — race condition if two concurrent PRs
  • ⚠ Heatmap is 1-week stale — real-time heatmap would be prohibitively expensive
  • ⚠ Privacy zones require all historical activities to be re-processed if zone changes

6. Detailed Design▼

Double-Entry Ledger (Core Invariant)

Every monetary movement creates exactly two ledger entries: one DEBIT and one CREDIT, summing to zero. This is the foundational accounting invariant. Implementation: INSERT INTO ledger_entries for both sides inside a single serializable DB transaction. Balance is derived by summing ledger entries (authoritative), but also maintained as a cached counter updated atomically with each write for fast reads. Account balance can never go negative for checking accounts without an explicit overdraft product linked. The ledger is append-only — mistakes are corrected with new reversal entries, never UPDATE or DELETE.

Payment Orchestration & Clearing Pipelines

Internal transfer: Single DB transaction — debit source, credit destination, both ledger entries, one payment_transactions row. Settles in milliseconds. ACH: NACHA batch format. Transactions accumulate across the day → batch file generated at 5 PM, submitted to Fed ACH → next-day settlement. Funds placed on hold immediately, released/reversed on settlement confirmation. Wire (Fedwire/CHIPS): Real-time gross settlement. Message sent to Fedwire within minutes, settled same-day. CHIPS nets multilateral obligations between major banks at end of day. SWIFT: ISO 20022 / MT103 messages routed through correspondent banking network. May take 1–3 days cross-border; GPI tracking added for real-time status.

Card Authorization (<100ms)

Visa/Mastercard sends ISO 8583 auth request to JPMC's card processor. Synchronous path: (1) Decrypt + parse message (HSM). (2) Lookup account in in-memory balance cache. (3) Parallel: fraud ML inference + OFAC check + velocity rule engine. (4) If approved: atomically decrement available_balance (place hold). (5) Return auth response. The entire path must complete in <100ms P99 — achieved by keeping balance cache fully in RAM (Redis/custom in-memory store), pre-loading ML models on GPU inference nodes, and running steps 3 in parallel. Settlement happens at end of day when the merchant submits the capture batch.

Fraud Detection Architecture

Two-layer system: (1) Real-time scoring (<50ms, inline with auth): gradient-boosted model + neural net ensemble. Features: transaction amount vs historical avg, merchant category, geo-velocity (impossible travel), device fingerprint, time-of-day pattern, beneficiary risk score. Model served from ONNX runtime on GPU inference pods. (2) Async behavioral analytics (minutes-to-hours): graph neural network on transaction graph to detect money mule networks, account takeover chains, bust-out fraud rings. Feature store (Redis + Cassandra) maintains rolling 90-day velocity counters per customer/card/device/IP. Fraud events trigger real-time push notification + optional account freeze.

AML & Sanctions Screening

Every outbound payment screens the beneficiary against OFAC SDN, UN Consolidated, EU Consolidated, and domestic FinCEN lists. Screening is synchronous and blocking for international wires — the payment cannot proceed on a hit. Algorithm: exact match on account number + IBAN; fuzzy phonetic match (Soundex + Jaro-Winkler) on beneficiary name; country risk scoring on beneficiary bank BIC. List updates pushed in real time from screening vendor (Accuity/WorldCheck). Suspicious Activity Reports (SARs) filed to FinCEN within 30 days of detection — automated SAR generation workflow for pattern-matched triggers.

High Availability: Active-Active Architecture

Payment critical path runs in two geographically separated datacenters simultaneously (active-active). Database uses synchronous replication (Oracle Data Guard with SYNC mode) — a write is only acknowledged when both DCs have committed. This guarantees zero data loss (RPO=0) at the cost of ~5ms cross-DC latency added to every write. Load balancer distributes 50/50 across DCs in normal operation. On DC failure: surviving DC takes 100% of traffic within 30 seconds (automated health-check + DNS failover). RTO target: 30 seconds. The <5 min/year downtime (99.999%) budget is consumed primarily by planned maintenance windows, not failures.

FX Rate Engine

Treasury desk receives live mid-rates from Reuters/Bloomberg via FIX protocol. FX service maintains in-memory rate cache (updated every 100ms). Retail customer rates = mid ± spread (spread varies by amount tier and customer segment). Quote API locks a rate for 30 seconds (quote_id stored in Redis with TTL). Booking converts at the locked rate regardless of subsequent market moves. Large FX trades (>$1M) routed to treasury trading desk for manual execution. All rate locks, quotes, and executions written to fx_rates table for audit and P&L reconciliation.

Design Choices

  • ✓ Append-only ledger — no UPDATE/DELETE ever, full auditability
  • ✓ Synchronous dual-DC replication — RPO=0, no data loss
  • ✓ In-memory balance cache — avoids DB hit on every auth (<1ms read)
  • ✓ Idempotency keys on all payment APIs — safe retries
  • ✓ Separate available_balance from current_balance — holds model
  • ✓ Parallel fraud + OFAC in card auth — latency budget preserved

Tradeoffs

  • ⚠ Synchronous replication adds 5ms cross-DC latency to every write
  • ⚠ Ledger-derived balance is slow — cached balance can diverge on crash
  • ⚠ ACH next-day settlement means holds must last 24h — customer UX impact
  • ⚠ SWIFT correspondent banking opacity — GPI helps but tracking still imperfect
  • ⚠ ML fraud models need constant retraining — adversarial drift is rapid
  • ⚠ Mainframe core banking (COBOL) — extremely reliable but very hard to evolve

Message Fan-Out: The Large Server Problem

500K-member server with ~2–5% online at any time = 10–25K connected users. Fan-out via pub/sub: message service publishes to topic per channel. Gateway nodes subscribe to topics for channels their connected users are in. Each gateway fans out to its own ~10K connections only. For mega-servers (1M+), selective delivery based on active viewport membership. Not all 500K get the push — only online sessions subscribed to that channel.

ScyllaDB + Snowflake IDs

channel_id as partition key means all messages for a channel are co-located. Snowflake message_id (64-bit: timestamp + worker + sequence) as clustering key DESC — messages naturally ordered, time-based pagination (before/after) needs no separate index. Discord migrated from Cassandra to ScyllaDB for lower P99 latency at their scale. Hot channels handled by Scylla's compaction. Snowflake IDs encode creation timestamp — range scans are physically sequential.

Voice: WebRTC SFU (not mesh)

Full mesh: N users × (N-1) streams = O(N²) bandwidth per user. SFU (Selective Forwarding Unit): each user sends 1 upstream to SFU; SFU selectively forwards to all N-1 receivers. O(N) total. Opus codec at 48kHz provides good quality at 32–128 Kbps. Discord deploys SFU nodes globally — each voice channel is pinned to the geographically closest SFU. Video screen sharing uses VP8/H264 simulcast layers.

  • ✓ Gateway fan-out to online users only — avoids 500K full delivery

Dispatch Engine: Nearest Rider Matching

On order creation: (1) Geocode pickup to lat/lng. (2) GEORADIUS (lat, lng, 3km) on Redis for all online riders with current_orders_count < max_concurrent. (3) Sort by distance + acceptance_rate + rating. (4) Offer to top candidate via push notification (30s timeout). (5) No response → offer to next candidate. (6) Acceptance → rider status = busy, order.rider_id set. Radius expansion: 3km → 5km → 10km → supervisor queue if no takers.

ETA Computation

ETA = rider-to-pickup travel time + 2min pickup buffer + pickup-to-drop travel time. Travel time from in-house ML model trained on historical Shadowfax delivery times by zone + time-of-day, or Google Distance Matrix API. Re-computed every 60s and updated in order. SSE pushes ETA update if deviation >2 min. India-specific: accounts for traffic congestion patterns, narrow lanes, and monsoon delays.

Design Choices

  • ✓ Redis GEORADIUS — sub-ms nearest-rider spatial query
  • ✓ Offer-and-timeout — simple, no distributed lock needed
  • ✓ Kafka decouples location ingest from persistence + tracking
  • ✓ OTP delivery confirmation — prevents false delivery fraud

Tradeoffs

  • ⚠ Sequential offer model — slow in low-supply areas (multiple 30s waits)
  • ⚠ GPS accuracy ±20m in dense Indian cities — floor-level ambiguity
  • ⚠ ETA degrades severely during rain, festivals, traffic jams
  • ⚠ COD cash handling at rider creates reconciliation complexity

Multi-Currency Pocket Model

Each user has N currency_pocket rows (one per currency held). FX exchange atomically debits EUR pocket, credits GBP pocket, and writes a transfer row — all in one SERIALIZABLE CockroachDB transaction distributed across EU+UK+US regions. The pocket model makes balance queries trivially simple (single row read) and keeps currencies fully isolated. Balances are always ≥ 0 (enforced at DB level). Reserved_balance tracks pending card auth holds separately from available balance.

Crypto Dealer Model

Revolut acts as dealer, not an exchange. User buys £100 BTC from Revolut's internal book at marked-up rate. Revolut simultaneously buys equivalent BTC from prime broker (Bitstamp/Coinbase Pro) at tighter spread — pocketing the difference. User gets instant <500ms execution; Revolut carries no directional exposure. Crypto positions are Revolut's internal ledger — not on-chain. On-chain withdrawal is a separate, slower flow. This model requires no blockchain interaction for 99% of trades.

Design Choices

  • ✓ CockroachDB — ACID + geo-distributed, no single region SPOF
  • ✓ Dealer model for crypto — instant UX, no on-chain latency
  • ✓ Idempotency keys — safe retries on network failures
  • ✓ Weekend spread markup — automatically hedges FX gap risk

Tradeoffs

  • ⚠ Dealer model = Revolut holds crypto exposure (counterparty risk)
  • ⚠ CockroachDB cross-region writes add ~5ms latency vs single-region PG
  • ⚠ FX 15s staleness — user may get worse rate than shown
  • ⚠ EMI license limits — deposits not FSCS/DGS insured in all regions

Concurrent Bid Conflict Resolution

The invariant: exactly one bidder wins at any moment. For hot auctions (10K bids/sec): Redis Lua script atomically compare-and-swaps the current price. If amount > current_price → update current_price + current_winner → return accepted. If lost race → return outbid. Redis is the authoritative real-time price; PostgreSQL is the durable record (async write-behind). For moderate load: PostgreSQL optimistic locking: UPDATE auctions SET current_price=?, version=version+1 WHERE auction_id=? AND version=? AND current_price < ?. Zero rows updated → conflict.

Anti-Sniping + Proxy Bidding

Anti-sniping: every bid checks if NOW() > end_time - anti_snipe_minutes. If true: extend end_time += anti_snipe_minutes. This extension happens atomically within the same Redis Lua script as the bid acceptance — it cannot be a separate operation. Proxy bidding: on proxy bid set, engine immediately computes minimum winning bid and places it. On any competing bid: proxy engine auto-bids to outbid if still below max. When two proxy bidders collide: earlier proxy wins at the later proxy's max + 1 minimum increment.

Design Choices

  • ✓ Redis Lua CAS — atomic bid acceptance, eliminates race conditions
  • ✓ Anti-snipe extension atomic with bid — cannot be bypassed
  • ✓ Proxy max AES-256 encrypted — never readable from logs or DB dumps
  • ✓ auction_events append-only — complete audit trail for disputes
  • ✓ Partition bid engine by auction_id — horizontal scale without cross-shard ops

Tradeoffs

  • ⚠ Anti-snipe makes close time non-deterministic — auction scheduler complexity
  • ⚠ Redis as authoritative price + async DB → data loss window on Redis crash
  • ⚠ Proxy bidding reveals max via incremental probing attacks
  • ⚠ Auction close is a hard deadline — distributed timer reliability critical
  • ⚠ Hot auction serial queue adds latency for all bidders, not just winner
7. Distinctive Component: Atomic Bid Engine + Anti-Sniping + Proxy Bidding▼

An auction platform's uniqueness is the bid engine: exactly one winner from 10K/sec concurrent last-second bids, anti-sniping extension atomic with each bid, real-time proxy outbidding, and live price updates to all watchers — all with sub-500ms latency and fairness users can trust.
