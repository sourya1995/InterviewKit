Synchronization Primitives
Core Building Blocks
Rule of thumb: prefer higher-level constructs (Channel, SemaphoreSlim, ConcurrentQueue) over raw lock/Monitor unless you need fine-grained control. Every primitive below has a cost — choose the lightest one that is correct.

Concurrency Patterns
14 Patterns
P→C
Producer / Consumer
Decoupled pipeline via shared buffer
Producers enqueue work; consumers dequeue and process independently. The buffer absorbs bursts and decouples throughput rates.
.NET Implementation
Channel<T> BlockingCollection<T> ConcurrentQueue<T>
Use when
  • Kafka message ingestion pipeline
  • Order processing queue (Amazon Shopping)
  • Log entry buffering (Logging System)
  • Ride request dispatch (Ride Sharing)
var channel = Channel.CreateBounded<Order>(1000); // Producer await channel.Writer.WriteAsync(order); // Consumer await foreach (var o in channel.Reader.ReadAllAsync()) { }
TPL
Thread Pool
Reuse threads — avoid creation overhead
Maintain a pool of reusable worker threads. Submit work items; the pool schedules them onto available threads. .NET's CLR thread pool backs Task.Run.
.NET Implementation
Task.Run() ThreadPool.QueueUserWorkItem Parallel.ForEach
T-01 BUSY
T-02 BUSY
T-03 IDLE
T-04 IDLE
T-05 BUSY
T-06 IDLE
CLR ThreadPool — auto-scales min/max workers
R/W
Reader-Writer Lock
Shared reads, exclusive writes
Allow concurrent reads but serialise writes. Optimal when reads far outnumber writes — avoids unnecessary exclusive locking on read paths.
.NET Implementation
ReaderWriterLockSlim ImmutableDictionary + Interlocked.Exchange
Use when
  • Library catalog — many searchers, rare updates
  • Stock price cache — frequent reads, periodic writes
  • ATM session store — read often, write on transaction
var rwls = new ReaderWriterLockSlim(); rwls.EnterReadLock(); try { /* read */ } finally { rwls.ExitReadLock(); }
ACT
Actor Model
Isolated state + message-passing
Each actor has private state and a mailbox. No shared memory — actors communicate only via async messages. Eliminates lock contention entirely.
.NET Implementation
Microsoft Orleans (Virtual Actors) Akka.NET TPL Dataflow ActionBlock
Use when
  • Elevator controller — each elevator is an actor
  • Chess game — each game session is isolated
  • Ride sharing — driver/rider state actors
PIPE
Pipeline (Dataflow)
Staged concurrent processing chain
Chain of processing stages where each stage consumes output of the previous. Stages run concurrently; bounded buffers between stages provide backpressure.
.NET Implementation
TPL Dataflow (TransformBlock) System.Threading.Channels IAsyncEnumerable pipeline
Use when
  • Kafka: Fetch → Deserialise → Process → Commit
  • Order: Validate → Reserve → Charge → Notify
  • Image upload: Decode → Resize → Store → CDN
var fetch = new TransformBlock<Req,Msg>(...); var process = new TransformBlock<Msg,Result>(...); var store = new ActionBlock<Result>(...); fetch.LinkTo(process); process.LinkTo(store);
F|J
Fork / Join
Divide work, run in parallel, aggregate results
Split a task into independent subtasks (fork), execute concurrently, then combine results (join). Classic map-reduce shape.
.NET Implementation
Task.WhenAll() Parallel.ForEach PLINQ (.AsParallel())
Use when
  • Airline: search multiple GDS providers in parallel
  • Amazon: fetch price, stock, reviews concurrently
  • Splitwise: compute balances for all group members
var results = await Task.WhenAll( FetchPriceAsync(id), FetchStockAsync(id), FetchReviewsAsync(id));
⟳
Async / Await (TAP)
Task-based Asynchronous Pattern
Non-blocking I/O without explicit thread management. The compiler rewrites async methods into state machines. Thread is released during awaits, improving scalability.
Key rules
  • Async all the way — never .Result / .Wait() on async code
  • Use ConfigureAwait(false) in library code
  • CancellationToken should propagate through the entire call chain
  • ValueTask for hot paths that often complete synchronously
// ✗ sync-over-async — deadlock risk var result = GetDataAsync().Result; // ✓ proper async chain var result = await GetDataAsync(ct);
CAS
Lock-Free / CAS
Optimistic concurrency via Compare-And-Swap
Read → compute → CAS atomically. If CAS fails (another thread changed the value), retry. No blocking, no deadlocks — but ABA problem can occur.
.NET Implementation
Interlocked.CompareExchange Interlocked.Increment/Add Volatile.Read/Write
Use when
  • Parking lot: atomic spot counter
  • ATM: account balance CAS update
  • Redis: key expiry tick counter
// Atomic increment — no lock needed Interlocked.Increment(ref _availableSpots); // CAS — update only if value still expected Interlocked.CompareExchange(ref _bal, newBal, expected);
SEM
Semaphore / Throttle
Limit concurrent access to N
Cap the number of concurrent operations to protect downstream systems, enforce rate limits, or model finite resource pools (DB connections, API quotas).
.NET Implementation
SemaphoreSlim(N, N) RateLimiter (.NET 7+) Polly — Bulkhead policy
Use when
  • Amazon Locker: N lockers = N concurrent assignments
  • Movie Booking: max 10 concurrent seat-hold transactions
  • Stock Brokerage: throttle external market data calls
var _sem = new SemaphoreSlim(10, 10); await _sem.WaitAsync(ct); try { await AssignLockerAsync(); } finally { _sem.Release(); }
SCH
Scheduler / Work Dispatcher
Assign & prioritise work across workers
Central dispatcher assigns tasks to workers based on priority, affinity, or load. Workers pull from a priority queue or dedicated queues.
.NET Implementation
PriorityQueue<T,P> ConcurrentPriorityQueue (custom) TaskScheduler subclass
Use when
  • Elevator: LOOK/SCAN scheduling algorithm
  • Ride sharing: driver assignment dispatcher
  • Meeting scheduler: room conflict resolver
IMM
Immutability
Eliminate shared mutable state entirely
Immutable objects can be safely shared across threads with zero synchronisation. Create new versions rather than mutating in place.
.NET Implementation
record / readonly struct ImmutableDictionary<K,V> ImmutableList / ImmutableArray System.Collections.Immutable
Use when
  • Chess: board state snapshots (undo / replay)
  • Stock brokerage: order book snapshots
  • Config / routing tables read by many threads
DCL
Double-Checked Locking
Lazy thread-safe singleton initialisation
Check before locking, then check again inside the lock to avoid unnecessary locking on the hot path after initialisation. In .NET, prefer Lazy<T> which handles this correctly.
.NET Implementation
Lazy<T>(LazyThreadSafetyMode.ExecutionAndPublication)
// Preferred — Lazy<T> handles DCL internally private static readonly Lazy<ParkingLot> _inst = new(() => new ParkingLot()); public static ParkingLot Instance => _inst.Value;
BP
Backpressure
Slow producers when consumers lag
When consumers can't keep up, signal producers to slow down rather than unbounded memory growth. Critical for stream processing systems.
.NET Implementation
Channel.CreateBounded(capacity) BoundedChannelFullMode.Wait BlockingCollection(boundedCapacity)
Use when
  • Kafka consumer groups — offset commit lag
  • Logging: drop or block when buffer full
  • Ride sharing: cap queued ride requests
OCC
Optimistic Concurrency
Version/timestamp-based conflict detection
Read data with a version token, compute update, write only if version unchanged (otherwise retry). Avoids locking under low-contention scenarios.
.NET / DB Implementation
EF Core — RowVersion / ConcurrencyToken Cosmos DB — ETag Redis WATCH / MULTI / EXEC
Use when
  • Seat booking — simultaneous seat selection
  • ATM — concurrent balance updates
  • Stock order — concurrent order modifications

async/await — The .NET Model
TAP + ValueTask + IAsyncEnumerable
TAP
Task Asynchronous Pattern
The standard .NET async model
  • Return Task for async void-equivalent; Task<T> for results
  • Always accept CancellationToken as last parameter
  • Suffix async methods with Async
  • Never use async void except in event handlers
public async Task<Booking> BookSeatAsync( int seatId, CancellationToken ct = default) { await _sem.WaitAsync(ct); try { return await _repo.BookAsync(seatId, ct); } finally { _sem.Release(); } }
VT
ValueTask<T>
Zero-alloc async for hot paths
When a method frequently completes synchronously (cache hit), ValueTask avoids heap allocation of a Task object. Used in Redis, SignalR, ASP.NET Core hot paths.
  • Redis GET — cache hit → synchronous ValueTask return
  • Channel.Reader.ReadAsync — often has data ready
  • ⚠ Never await ValueTask twice — not reusable
public ValueTask<string?> GetCachedAsync(string key) => _cache.TryGetValue(key, out var v) ? ValueTask.FromResult(v) // zero alloc : new(FetchFromDbAsync(key)); // async path
IAE
IAsyncEnumerable<T>
Streaming async sequences
Pull-based async stream. Consumer controls pace (backpressure). Ideal for database cursor reads, Kafka topic streaming, and paginated API results.
  • Cricinfo: stream ball-by-ball commentary
  • Kafka: stream messages from partition
  • File system: stream directory entries
public async IAsyncEnumerable<Ball> StreamBallsAsync( int matchId, [EnumeratorCancellation] CancellationToken ct) { await foreach (var b in _db.GetBallsAsync(matchId, ct)) yield return b; }
CT
CancellationToken Pattern
Cooperative cancellation through the chain
Cancellation must be cooperative — pass the token to every async call. Link tokens to create timeouts. Register callbacks for cleanup.
// Combine user cancel + timeout using var cts = CancellationTokenSource .CreateLinkedTokenSource(userCt, timeoutCt); using var timeout = new CancellationTokenSource(TimeSpan.FromSeconds(30)); var linkedCt = CancellationTokenSource .CreateLinkedTokenSource(userCt, timeout.Token).Token;

Concurrency Anti-Patterns
What NOT to do
Sync-over-Async (.Result / .Wait())
Calling .Result or .Wait() on a Task in async context will deadlock in environments with a SynchronizationContext (ASP.NET Classic, WinForms, WPF). Thread pool thread is blocked while waiting for a continuation that needs the same thread.
Async all the way. If you must bridge, use Task.Run(() => AsyncMethod()).Result in a threadpool context only.
async void (outside event handlers)
Exceptions from async void methods cannot be caught by the caller and will crash the process. The caller has no way to await completion or observe the result.
Always return Task or Task<T>. Only exception: UI event handlers (Button.Click) where the signature is forced.
Captured Mutable Closures in Parallel Loops
Capturing a loop variable (e.g. i) in a lambda passed to Parallel.ForEach or Task.Run creates a race — all tasks may see the final value of i.
Copy the variable before capture: var local = i; then use local inside the lambda.
lock() on public / string / Type objects
Locking on publicly visible objects (this, typeof(T), or string literals) allows outside code to take the same lock, causing unexpected deadlocks.
Always lock on a private readonly object _lock = new(); that no external code can reference.
Nested / Inconsistent Lock Ordering
Thread A acquires lockX then lockY; Thread B acquires lockY then lockX. Classic deadlock. Very common in Account transfer scenarios (ATM, Splitwise).
Always acquire multiple locks in a globally consistent order (e.g. by ID). Or eliminate nested locking entirely by restructuring.
Unbounded Parallelism (Task.Run in a loop)
Launching unlimited tasks in a loop floods the thread pool, causes context-switch thrashing, and can starve other work. Common mistake in bulk processing.
Use SemaphoreSlim to cap concurrency, Parallel.ForEachAsync with MaxDegreeOfParallelism, or TPL Dataflow with BoundedCapacity.
Forgotten ConfigureAwait(false) in Libraries
In library code, awaiting without ConfigureAwait(false) captures the caller's SynchronizationContext. This forces continuations back to the UI/ASP.NET thread, reducing scalability and risking deadlocks.
In all library/infrastructure code, always use .ConfigureAwait(false). In application-layer code it's optional but consistent usage is cleaner.
ConcurrentDictionary Non-Atomic Compound Ops
Reading then writing to ConcurrentDictionary in two separate calls is NOT atomic. Between your TryGetValue and subsequent update, another thread can modify the value.
Use AddOrUpdate or GetOrAdd with the factory overload. For custom compound logic, use Interlocked.CompareExchange on a reference or lock on the specific key.
Thread.Sleep in Production Code
Thread.Sleep blocks the OS thread, wasting thread pool resources. In async code it makes no sense. In tight retry loops it can cause thread pool starvation.
Use await Task.Delay(ms, ct) for async waits. Use Polly for retry logic with exponential back-off and jitter.
