GC Internals & Tuning

1.1 What Actually Causes GC Pauses in Production
GClatency
▼
Gen 0/1 collections are sub-millisecond and triggered frequently. Gen 2 and LOH (Large Object Heap) collections are stop-the-world and are what causes latency spikes. A Gen 2 collection fires when Gen 1 survivors fill Gen 2 — typically because short-lived objects live long enough to be promoted (the "mid-life crisis" problem: large request-scoped collections, deserialised DTOs stored in caches).

The LOH holds allocations ≥85,000 bytes and is not compacted by default — fragmentation grows over time. In .NET 5+, trigger compaction explicitly. Server GC (default in ASP.NET Core) uses one heap per logical CPU for higher throughput but higher baseline memory. Monitor gen-2-gc-count and loh-size with dotnet-counters before reaching for GC knobs.
gc-tuning.cs
// SustainedLowLatency — suppresses Gen 2 for time-critical sections.
// Use only in bounded critical sections; always restore in finally.
GCSettings.LatencyMode = GCLatencyMode.SustainedLowLatency;
try   { /* real-time work */ }
finally { GCSettings.LatencyMode = GCLatencyMode.Interactive; }

// Force LOH compaction before next Gen 2 — use sparingly
GCSettings.LargeObjectHeapCompactionMode =
    GCLargeObjectHeapCompactionMode.CompactOnce;
GC.Collect(GC.MaxGeneration, GCCollectionMode.Forced, blocking: true);
Interview signal Be able to explain why promoting an object from Gen 0 to Gen 2 is more expensive than just the extra collection cycle — it also increases GC roots, extends mark phase, and can fragment the heap.
1.2 Finalizers — The Two-Cycle Penalty and SafeHandle
GCalloc
▼
Objects with finalizers cannot be collected in their first eligible GC. The GC promotes them to the finalization queue; a dedicated finalizer thread runs the destructor, then the object is collected in the next cycle — two full GC cycles per finalizable object. Under load, the finalizer queue can back up, delaying collection indefinitely.

The fix: implement IDisposable, call GC.SuppressFinalize(this) in Dispose(). Only add a finalizer if you hold unmanaged resources directly — which is rare since SafeHandle wraps OS handles and provides critical finalization correctly.
safe-handle.cs
// Prefer SafeHandle — runtime manages finalization; no raw finalizer needed
public class NativeBuffer : SafeHandle
{
    public NativeBuffer(int size) : base(IntPtr.Zero, ownsHandle: true)
        => SetHandle(Marshal.AllocHGlobal(size));

    public override bool IsInvalid => handle == IntPtr.Zero;

    protected override bool ReleaseHandle()
    {
        Marshal.FreeHGlobal(handle);
        return true;
    }
    // Critical finalizer provided by SafeHandle — no ~NativeBuffer() needed
}
1.3 Object Pooling — ObjectPool<T> and RecyclableMemoryStream
GCalloc
▼
For objects expensive to construct and frequently recycled (parsers, serializers, StringBuilder), an object pool eliminates the alloc + GC cycle entirely. ASP.NET Core's ObjectPool<T> (from Microsoft.Extensions.ObjectPool) is the standard approach.

For streams: Microsoft.IO.RecyclableMemoryStream eliminates LOH allocations by chunking large streams into pooled small buffers — critical for services that serialize large payloads per-request.
object-pool.cs
// Register — pool is singleton, instances are transient
services.AddSingleton<ObjectPool<StringBuilder>>(sp =>
    sp.GetRequiredService<ObjectPoolProvider>()
      .CreateStringBuilderPool());

public class ReportService(ObjectPool<StringBuilder> pool)
{
    public string Build(IEnumerable<Row> rows)
    {
        var sb = pool.Get();
        try   { foreach (var r in rows) sb.AppendLine(r.ToString()); return sb.ToString(); }
        finally { pool.Return(sb); }   // policy calls sb.Clear()
    }
}

Allocation Strategies

2.1 Span<T>, Memory<T>, and the Ownership Model
alloczero-copy
▼
Span<T> is a ref struct — compiler-enforced to live on the stack only. It cannot be stored in fields, boxed, or cross await boundaries. This constraint is what makes it safe over stack-allocated and unmanaged memory.

Memory<T> is the heap-compatible counterpart: storable in fields, safe across await, sliceable to Span<T> at call sites via .Span. Use Memory<T> in async pipelines, Span<T> in synchronous hot paths.

IMemoryOwner<T> adds lifecycle: the owner Dispose()s the memory back to a pool. This is the pattern used internally by System.IO.Pipelines — zero-copy I/O where data is written directly into memory the reader examines without a copy. Kestrel uses this for HTTP/2 framing.
memory-ownership.cs
// Span — synchronous only, stack-constrained, zero alloc
static int ParseFirstInt(ReadOnlySpan<char> text)
{
    int sep = text.IndexOf(',');
    return int.Parse(sep > -1 ? text[..sep] : text);
}

// Memory — async-safe, can be stored and awaited
async ValueTask ProcessAsync(IMemoryOwner<byte> owner)
{
    await socket.SendAsync(owner.Memory, SocketFlags.None);
    owner.Dispose();   // returns to pool
}

// System.IO.Pipelines — zero-copy network I/O at scale
var pipe = new Pipe();
await Task.WhenAll(FillPipeAsync(socket, pipe.Writer),
                   ReadPipeAsync(pipe.Reader));
2.2 Struct Layout — Cache Lines and False Sharing
allocCPU cache
▼
The CLR pads struct fields to natural alignment. A bool, long, bool struct consumes 24 bytes (not 10) due to padding. For hot data structures iterated in tight loops, this wastes cache lines and slows throughput.

More critically: in multi-threaded code, two threads writing to fields that share a 64-byte cache line cause false sharing — constant cache-line invalidation across cores. Pad hot concurrent structs to exactly 64 bytes to isolate them.
struct-layout.cs
// ❌ 24 bytes: bool(1)+pad(7)+long(8)+bool(1)+pad(7)
struct Bad { bool A; long B; bool C; }

// ✅ Reorder by descending size — 10 bytes, no wasted padding
struct Good { long B; bool A; bool C; }

// ✅ Pad to 64 bytes to prevent false sharing across CPU cores
[StructLayout(LayoutKind.Explicit, Size = 64)]
struct PaddedCounter
{
    [FieldOffset(0)] public long Value;
    // Remaining 56 bytes padding — each counter isolated to its own cache line
}
2.3 Source Generators — Eliminating Reflection Overhead at Runtime
allocAOT
▼
Reflection-based serialization uses MethodInfo.Invoke — boxes value types, allocates parameter arrays, prevents JIT inlining. Source generators (System.Text.Json [JsonSerializable], Mapster codegen) emit direct property access at compile time: zero reflection, JIT-friendly, NativeAOT-compatible.

Critical for cold-start performance in Lambda/serverless and for services with strict startup SLAs.
source-gen-json.cs
// Compiled at build time — zero reflection at runtime
[JsonSerializable(typeof(OrderDto))]
[JsonSerializable(typeof(List<OrderDto>))]
[JsonSourceGenerationOptions(PropertyNamingPolicy = JsonKnownNamingPolicy.CamelCase)]
internal partial class AppJsonContext : JsonSerializerContext { }

// Usage — uses generated type metadata, not reflection
string json  = JsonSerializer.Serialize(order,  AppJsonContext.Default.OrderDto);
OrderDto dto = JsonSerializer.Deserialize(json, AppJsonContext.Default.OrderDto);

Async at Scale

3.1 State Machine Overhead — When async Costs More Than It Saves
asyncalloc
▼
The C# compiler transforms every async method into a state machine struct. When the method actually suspends (hits an incomplete await), the struct is boxed onto the heap. When it completes synchronously, no boxing occurs — but the state machine overhead still exists.

Implication: marking a method async that rarely suspends pays the state machine cost on every call for a capability rarely used. Prefer returning completed tasks or ValueTask for methods that are synchronous on the hot path. Avoid thin wrapper async methods that just await and return — pass the Task directly.
async-cost.cs
// ❌ Unnecessary state machine — never suspends, but pays the cost
public async Task<int> GetCountAsync() => _items.Count;

// ✅ Zero overhead — completed Task, no state machine
public Task<int> GetCountAsync() => Task.FromResult(_items.Count);

// ✅ ValueTask — zero alloc on sync path, alloc only on actual async path
public ValueTask<User> GetUserAsync(int id) =>
    _cache.TryGetValue(id, out var u)
        ? new ValueTask<User>(u)            // sync — no alloc
        : new ValueTask<User>(FetchAsync(id)); // async — alloc once
ValueTask caution Never await a ValueTask more than once. Never store it across await boundaries or use it in Task.WhenAll. Convert to Task with .AsTask() if you need to fan-out.
3.2 Thread Pool Starvation — Diagnosing Root Causes
asynccontention
▼
Thread pool starvation occurs when all pool threads are blocked — typically by synchronous waits over async I/O. The pool starts with ProcessorCount threads and injects one new thread per second when demand exceeds supply. Starvation can take tens of seconds to self-heal; during that time, latency spikes and requests queue up while CPU utilisation looks deceptively low.

Root causes in production: (1) .Result / .Wait() in middleware or services, (2) Task.Run() flooding the pool with CPU-bound work, (3) SemaphoreSlim.Wait() instead of WaitAsync(). Diagnose with dotnet-counters — watch for ThreadPool Queue Length and Thread Count rising together.
starvation.sh
# Watch live — starvation: queue grows faster than thread count
dotnet-counters monitor -p <PID> \
  System.Runtime[threadpool-queue-length] \
  System.Runtime[threadpool-thread-count]

# Capture dump during starvation — look for blocked threads
dotnet-dump collect -p <PID>
dotnet-dump analyze <dump> -c "clrthreads -live" -c "syncblk"
3.3 CancellationToken Propagation — Correctness and Resource Recovery
async
▼
Missing token propagation means cancelled HTTP requests still execute their full pipeline — hitting the database, allocating memory, and computing results for a response that will be discarded. At scale, this wastes thread pool threads and database connections on abandoned work.

Pass the token at every I/O boundary: EF queries, HttpClient calls, Channel reads, SemaphoreSlim.WaitAsync. Combine with a per-operation timeout using CreateLinkedTokenSource.
cancellation.cs
public async Task<Report> GenerateAsync(CancellationToken ct)
{
    // Combine caller cancellation with a hard 5-second timeout
    using var cts = CancellationTokenSource.CreateLinkedTokenSource(ct);
    cts.CancelAfter(TimeSpan.FromSeconds(5));

    var data = await _db.Orders
        .AsNoTracking()
        .Where(o => o.Date > DateTime.UtcNow.AddDays(-30))
        .ToListAsync(cts.Token);          // token cancels SQL command mid-flight

    var enriched = await _client.EnrichAsync(data, cts.Token); // cancels HTTP call

    return BuildReport(enriched);
}

LINQ & Data Access

4.1 IEnumerable vs IQueryable — The Silent Full-Table Scan
querySQL
▼
IQueryable.Where() builds an expression tree translated to a SQL WHERE clause. IEnumerable.Where() runs a delegate in-process — after materialising the full result set. A single AsEnumerable() mid-chain silently pulls every row from the database before your filter runs.

This is the most common performance bug found in repository patterns: exposing IEnumerable<T> from a repository method forces callers to load all rows. Expose IQueryable<T> so callers compose their filters before materialisation.
queryable.cs
// ❌ AsEnumerable() materialises ALL rows, then filters in-process
var result = _db.Orders
    .AsEnumerable()                  // ← full table load here
    .Where(o => o.Total > 1000)     // ← runs in-process on ALL rows
    .ToList();

// ✅ Filter stays in SQL — single round-trip
var result = await _db.Orders
    .Where(o => o.Total > 1000)
    .AsNoTracking()
    .ToListAsync(ct);

// ❌ Repository returning IEnumerable — caller loads all rows
public IEnumerable<Order> GetOrders() => _db.Orders;

// ✅ Repository returning IQueryable — caller composes, one SQL generated
public IQueryable<Order> GetOrders() => _db.Orders.AsNoTracking();
4.2 LINQ in Hot Paths — The Delegate and Iterator Tax
LINQhot path
▼
In-memory LINQ overhead comes from: iterator state machines (heap allocation per chain step), delegate invocations (no cross-site inlining), and inability to use Span<T>. For in-memory collections called >100k times/second, explicit for loops with direct indexing are 3–10× faster.

PLINQ pitfall: .AsParallel() on collections under ~10,000 items is typically slower than sequential LINQ due to partitioning and synchronization overhead. Profile with BenchmarkDotNet before reaching for PLINQ.
Don't speculate Never drop LINQ for raw loops without benchmarking. The JIT can often inline and vectorise LINQ chains at steady state. Measure first — the crossover point depends on collection size, chain complexity, and whether the data fits in L1/L2 cache.

EF Core Deep Dive

5.1 Change Tracker Cost — Set NoTracking Globally
EFalloc
▼
The change tracker maintains a snapshot of original values for every tracked entity (EntityEntry, PropertyEntry objects). A 1,000-row query creates 1,000 entity entries plus property snapshots — all heap-allocated and alive until the DbContext is disposed. For read-heavy APIs, this is pure overhead.

Best practice: set QueryTrackingBehavior.NoTracking globally on the DbContext options and opt back in with .AsTracking() only for write paths.
ef-notracking.cs
// Set globally — flips the default for all queries on this context
services.AddDbContext<AppDbContext>(opts =>
    opts.UseSqlServer(conn)
        .UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking));

// Opt in for write operations only
var order = await _db.Orders
    .AsTracking()
    .FirstAsync(o => o.Id == id, ct);

order.Status = OrderStatus.Shipped;
await _db.SaveChangesAsync(ct);
5.2 EF Core 7+ Bulk Mutations — ExecuteUpdate/ExecuteDelete
EFbulk
▼
Pre-EF7, updating 10,000 rows required loading them all, modifying in-memory, then emitting 10,000 individual UPDATE statements in SaveChanges. EF Core 7 introduced ExecuteUpdateAsync and ExecuteDeleteAsync — set-based SQL operations that translate to a single UPDATE ... WHERE or DELETE ... WHERE. No entity materialisation, no change tracker involvement.
ef-bulk.cs
// ❌ Pre-EF7: loads all rows + emits N UPDATE statements
var stale = await _db.Sessions.Where(s => s.LastActivity < cutoff).ToListAsync(ct);
foreach (var s in stale) s.IsExpired = true;
await _db.SaveChangesAsync(ct);

// ✅ EF Core 7+: single SQL UPDATE — no materialisation
await _db.Sessions
    .Where(s => s.LastActivity < cutoff)
    .ExecuteUpdateAsync(s => s.SetProperty(x => x.IsExpired, true), ct);

// ✅ Single SQL DELETE
await _db.AuditLogs
    .Where(l => l.CreatedAt < DateTime.UtcNow.AddYears(-1))
    .ExecuteDeleteAsync(ct);
5.3 Cartesian Explosion and AsSplitQuery
EF
▼
Including multiple collection navigations in a single query generates JOINs that produce a Cartesian product. 100 orders × 5 items × 3 tags = 1,500 rows transmitted from SQL for 100 logical entities. AsSplitQuery() issues separate SELECTs per collection, eliminating the explosion at the cost of multiple round-trips (and non-atomic reads).
Single Query One round-trip. Cartesian product multiplies with each collection. Best when collections are small.
Split Query N round-trips. No row multiplication. Not atomic — beware consistency under concurrent writes.
Manual Projection Maximum control. Select(new DTO) pulls only needed columns. Best for read-heavy DTO APIs.

Threading & Lock Contention

6.1 Contention — Detection and Elimination Strategies
contentionlatency
▼
A lock under high concurrency causes threads to queue in the OS kernel — each context switch is ~1–5µs. Under thousands of concurrent requests, a single hot lock serialises throughput entirely. Escalation path:

(1) Reduce lock scope — move computation outside the lock, lock only the mutation. (2) Reader/writer separation — ReaderWriterLockSlim for read-dominant workloads. (3) Lock-free — Interlocked for counters, ConcurrentDictionary with GetOrAdd. (4) Sharding — partition state into N independent locks, each thread hashes to its shard.
lock-strategies.cs
// ❌ Lock held during expensive computation — serialises all threads
lock (_sync) { result = ExpensiveCompute(); _cache[key] = result; }

// ✅ Compute outside lock; lock only the write
var result = ExpensiveCompute();
lock (_sync) { _cache.TryAdd(key, result); }

// ✅ Lock-free atomic increment — no kernel involvement
Interlocked.Increment(ref _requestCount);

// ✅ ConcurrentDictionary.GetOrAdd — striped internal locks
var val = _dict.GetOrAdd(key, k => new LazyLoader(k));
// ⚠ Factory may run on multiple threads if key absent simultaneously;
// wrap value in Lazy<T> if construction must be exactly once.
6.2 Channel<T> — Backpressure-Aware Async Pipelines
concurrencyasync
▼
System.Threading.Channels is the modern replacement for BlockingCollection and manual queue-plus-lock patterns. Bounded channels exert backpressure on producers when the consumer falls behind, preventing unbounded memory growth under load — critical for ingestion pipelines.
channels.cs
var channel = Channel.CreateBounded<WorkItem>(
    new BoundedChannelOptions(1000)
    {
        FullMode     = BoundedChannelFullMode.Wait, // backpressure on producer
        SingleWriter = false,
        SingleReader = true   // enables internal reader-path optimisation
    });

// Producer — awaits if channel is full (backpressure)
await channel.Writer.WriteAsync(item, ct);

// Consumer — IAsyncEnumerable, completes when writer closes
await foreach (var item in channel.Reader.ReadAllAsync(ct))
    await ProcessAsync(item, ct);

.NET Profiling Strategy

7.1 Choosing the Right Tool for the Symptom
profiling
▼
The senior mistake is jumping straight to a profiler on a dev box. Start with metrics in production to confirm the hypothesis, then scope with tracing, then isolate with benchmarking.
High CPU / Slow endpoint dotnet-trace CPU sampling → SpeedScope flamegraph. Or JetBrains dotTrace for wall-clock timeline with call trees.
GC pauses / Memory growth dotnet-counters live + dotnet-dump heap snapshot. PerfView or dotMemory for retained-root analysis.
Micro-benchmark validation BenchmarkDotNet — median/P99, auto-warmup, [MemoryDiagnoser] reports allocs/invocation. Always measure before and after.
dotnet-profiling.sh
# CPU flamegraph — 10 sec trace → open in SpeedScope
dotnet-trace collect -p <PID> --profile cpu-sampling --duration 00:00:10
dotnet-trace convert trace.nettrace --format Speedscope

# Live GC + thread pool counters — negligible overhead
dotnet-counters monitor -p <PID> System.Runtime

# Allocation hot-path trace
dotnet-trace collect -p <PID> --profile gc-verbose --duration 00:00:10
Interview signal "I confirm the symptom with dotnet-counters in production, capture a flamegraph with dotnet-trace to identify the hot path, write a targeted BenchmarkDotNet test to validate the fix in isolation, then ship and verify the metric drops." This demonstrates systematic diagnosis, not speculative tuning.

.NET Senior Performance Checklist

— Architecture & Code Review Level
senior
▼
  • Monitor Gen 2 GC count and LOH size in production with dotnet-counters before any GC config changes.
  • Never call .Result, .Wait(), or .GetAwaiter().GetResult() on a Task in request-handling code — risk of thread pool starvation and deadlock.
  • Use SafeHandle for unmanaged resources — avoids raw finalizers and their two-cycle GC penalty.
  • Use ObjectPool<T> or RecyclableMemoryStream for frequently constructed objects and large buffers.
  • Use Memory<T> in async pipelines; Span<T> in synchronous hot paths — never cross await with Span.
  • Consider System.IO.Pipelines for high-throughput stream processing — eliminates double-buffering used by Kestrel internally.
  • Use source-generated JSON serialization ([JsonSerializable]) in hot paths — eliminates reflection, NativeAOT-compatible.
  • Return ValueTask from methods that are synchronous on the hot path. Never await a ValueTask more than once.
  • Avoid async on methods that never suspend — return Task.FromResult() or a completed ValueTask directly.
  • Propagate CancellationToken to all I/O boundaries. Use CreateLinkedTokenSource for per-operation timeouts.
  • Set QueryTrackingBehavior.NoTracking globally on read-heavy DbContexts; opt in with .AsTracking() for writes.
  • Use ExecuteUpdateAsync / ExecuteDeleteAsync (EF Core 7+) for bulk mutations — no entity materialisation.
  • Expose IQueryable<T> from repositories, not IEnumerable<T> — keeps filter composition in SQL.
  • Avoid calling .AsEnumerable() mid-LINQ-chain on EF queries — it materialises the full result set.
  • Evaluate AsSplitQuery() when including multiple large collection navigations.
  • Move expensive computation outside lock scope; lock only the state mutation.
  • Use Channel<T> bounded with FullMode.Wait for producer/consumer pipelines — built-in backpressure.
  • Pad hot concurrent structs to 64 bytes to prevent false sharing across CPU cache lines.
  • Profile with BenchmarkDotNet ([MemoryDiagnoser]) before and after any micro-optimisation. Never speculate.

Reconciler & Fiber Internals

1.1 How Fiber Works — Interruptible, Prioritised Reconciliation
fiberrender
▼
React Fiber is a singly-linked tree of fiber nodes, each representing a unit of work. Unlike the stack-based reconciler (React 15), Fiber can pause, resume, reuse, or abort work mid-tree. Reconciliation splits into two phases:

Render phase (pure, async-interruptible) — runs component functions, builds the work-in-progress tree. Can be paused for higher-priority work (user input). May run multiple times per commit in Strict Mode — never put side effects in render.

Commit phase (synchronous, always runs to completion) — flushes DOM mutations, runs useLayoutEffect, then useEffect. Understanding this split is the foundation for reasoning about useTransition, useDeferredValue, and Suspense.
Lane model Each state update is enqueued with a priority lane. User input gets SyncLane (highest). useTransition updates get TransitionLane (low). The scheduler processes higher lanes first — the render phase can be interrupted mid-tree, the current work-in-progress thrown away, and a higher-priority update processed instead.
1.2 Reconciliation Heuristics — Keys, Types, and Intentional Remounts
fiberrender
▼
React's O(n) diffing relies on two assumptions: (1) Elements of different types produce different trees — React unmounts rather than reconciling across type boundaries. (2) Keys identify stable elements across renders.

Index keys on sortable/filterable lists cause React to reuse DOM nodes for different data — incorrect state retention, stale refs, and unnecessary re-renders. Use stable, data-derived keys.

Intentional key change is a valid pattern: changing a key forces a full unmount and remount, resetting all state and refs cleanly — often cleaner than multi-effect reset sequences.
keys.jsx
// ❌ Index key on sortable list — stale state retained after sort
{items.map((item, i) => <Row key={i} item={item} />)}

// ✅ Stable data key — reconciler correctly skips unchanged nodes
{items.map(item => <Row key={item.id} item={item} />)}

// ✅ Intentional remount to reset all state — cleaner than multi-effect reset
<UserProfileForm key={selectedUserId} userId={selectedUserId} />

React 18 Concurrent Features

2.1 useTransition — Interruptible Low-Priority Updates
concurrentrender
▼
useTransition marks a state update as non-urgent (TransitionLane). React starts rendering it at low priority. If a higher-priority update arrives (user keystroke), React abandons the in-progress transition render (throws away the work-in-progress tree), handles the urgent update, then restarts the transition. The UI always feels responsive.

This is distinct from debouncing: debouncing delays when work starts; transitions let React start the work immediately but interrupt it if something more important arrives.
use-transition.jsx
const [isPending, startTransition] = useTransition();

function handleChange(e) {
    setQuery(e.target.value);                // SyncLane — updates input immediately
    startTransition(() => {
        setResults(filterItems(e.target.value)); // TransitionLane — interruptible
    });
}

// isPending: true while transition is in-flight — show stale/dimmed content
return (
    <>
        <input value={query} onChange={handleChange} />
        <ResultList items={results} dim={isPending} />
    </>
);
2.2 useDeferredValue + memo — Defer Without Owning the Setter
concurrentrender
▼
useDeferredValue is the consumer-side equivalent of transitions — for props you don't control. During an urgent render, it returns the previous value; the new value is scheduled at low priority. Combine with React.memo: the deferred value doesn't change during the urgent render, so the memo comparison skips the expensive subtree entirely — the slow component simply isn't re-rendered until the deferred render fires.
deferred-value.jsx
function SearchPage({ query }) {
    const deferredQuery = useDeferredValue(query);

    // During urgent render: deferredQuery === prev value → memo skips ResultList
    // After deferred render fires: deferredQuery === new value → ResultList re-renders
    return (
        <>
            <SearchInput value={query} />
            <MemoizedResultList query={deferredQuery} />
        </>
    );
}

const MemoizedResultList = React.memo(ResultList);
// Comparison: prev deferredQuery === next deferredQuery
// During urgent render → same → expensive subtree skipped entirely
2.3 Automatic Batching — React 18 Breaking Behaviour Change
concurrentrender
▼
React 17 batched updates only inside React event handlers. Updates inside setTimeout, Promises, and native event listeners triggered separate renders. React 18 with createRoot batches all updates regardless of origin — a single render per event-loop tick. This is a breaking change for code relying on sequential renders between updates. Use flushSync to opt out for DOM measurement between two state sets.
batching.jsx
// React 18 — both setStates batched → one render (was two in React 17)
setTimeout(() => {
    setCount(c => c + 1);
    setFlag(true);
    // single re-render
}, 0);

// Opt out — flushSync forces synchronous render between calls
import { flushSync } from 'react-dom';

flushSync(() => setCount(c => c + 1));   // renders synchronously here
// read DOM layout...
flushSync(() => setFlag(true));           // renders synchronously here
03

Re-render Root Causes

3.1 Referential Instability — The Invisible Performance Leak
renderfiber
▼
React.memo uses shallow equality. Object literals, array literals, and inline functions are new references on every render — defeating memo entirely. A single context provider value re-creating an object literal per render cascades re-renders to every consumer in the entire tree.

This is the most common cause of unexpected re-render regressions in large React codebases. The fix is referential stability: module-level constants for truly static data, useMemo for dynamic objects, useCallback for callbacks passed to memoized children.
referential-stability.jsx
// ❌ New object reference every render — memo on Chart is useless
<Chart options={{ color: 'blue', grid: true }} />

// ✅ Module-level constant — stable reference forever
const CHART_OPTIONS = { color: 'blue', grid: true };
<Chart options={CHART_OPTIONS} />

// ❌ Context value re-creates on every AuthProvider render
<AuthCtx.Provider value={{ user, login, logout }}>

// ✅ Memoized — consumers only re-render when user/login/logout changes
const value = useMemo(() => ({ user, login, logout }), [user, login, logout]);
<AuthCtx.Provider value={value}>
3.2 Context Propagation — Split by Frequency, Not Domain
render
▼
Context triggers a re-render in every consumer when any value it holds changes — React performs no property-level diffing inside context values. The correct split is by update frequency, not domain.

A UserContext holding both the user object (changes rarely) and a cart counter (changes per interaction) re-renders every user-aware component on every cart update. Split by frequency: UserIdentityContext and CartContext.

For high-frequency shared state consumed by many components at different tree depths, Zustand or Jotai outperform Context — they use subscription-based updates that re-render only components subscribed to the specific slice that changed.

State Architecture at Scale

4.1 React Query Internals — Structural Sharing and Select
server statearchitecture
▼
Multiple components calling useQuery with the same key share a single cache entry — automatic request deduplication. On mount, a stale query refetches in the background while consumers render stale data immediately (stale-while-revalidate).

Structural sharing: React Query deep-compares responses and returns the same object reference for unchanged data. Components only re-render if their specific slice changed — even if the full query refetched.

select option: subscribe to a derived slice — the component re-renders only when that derived value changes, not on every query update.
react-query.jsx
// Select — re-renders only when cart item count changes (not full cart object)
const { data: count } = useQuery({
    queryKey: ['cart'],
    queryFn:  fetchCart,
    select: cart => cart.items.length,
});

// Optimistic update — update cache immediately, revert on error
const mutation = useMutation({
    mutationFn: updateItem,
    onMutate: async (updated) => {
        await queryClient.cancelQueries({ queryKey: ['cart'] });
        const prev = queryClient.getQueryData(['cart']);
        queryClient.setQueryData(['cart'], old =>
            ({ ...old, items: old.items.map(i => i.id === updated.id ? updated : i) })
        );
        return { prev };
    },
    onError:   (_, __, ctx) => queryClient.setQueryData(['cart'], ctx.prev),
    onSettled: ()           => queryClient.invalidateQueries({ queryKey: ['cart'] }),
});
4.2 useReducer + Dispatch Context — Stable References Without useCallback
renderarchitecture
▼
useReducer's dispatch has a stable identity across renders — React guarantees it never changes. Passing dispatch through context or to memoized children requires no useCallback. The canonical pattern: one context for state (changes frequently), one context for dispatch (never changes). Components that only dispatch actions don't re-render when state changes.
reducer-context.jsx
const StateCtx    = createContext(null);
const DispatchCtx = createContext(null);  // stable — never causes re-render

function StoreProvider({ children }) {
    const [state, dispatch] = useReducer(reducer, initialState);
    return (
        <DispatchCtx.Provider value={dispatch}>
            <StateCtx.Provider value={state}>{children}</StateCtx.Provider>
        </DispatchCtx.Provider>
    );
}

// ActionButton only consumes dispatch — never re-renders when state changes
const ActionButton = React.memo(() => {
    const dispatch = useContext(DispatchCtx);
    return <button onClick={() => dispatch({ type: 'INCREMENT' })}>+</button>;
});

Bundle & Loading Strategy

5.1 Module Federation — Shared Singletons in Micro-frontends
bundlearchitecture
▼
In a micro-frontend setup (Module Federation, single-spa), React and ReactDOM must be declared as shared singletons. Without this, each micro-frontend bundles its own React copy — causing hook constraint violations (hooks require a single React instance), broken context across app boundaries, and duplicate bundle weight.
module-federation.js
// webpack.config.js — host declares React as a shared singleton
new ModuleFederationPlugin({
    name: 'shell',
    shared: {
        react:        { singleton: true, requiredVersion: deps.react },
        'react-dom':  { singleton: true, requiredVersion: deps['react-dom'] },
        // Version mismatch → remote falls back to own copy, logs warning
        // strictVersion: true → throws on mismatch instead
    },
});
5.2 Prefetch and Preload — Eliminating Lazy-Load Waterfalls
bundleLCP
▼
Code splitting introduces a waterfall: main bundle loads → executes → discovers lazy import → network request for chunk. Prefetching eliminates this by loading chunks during idle time before they're needed. Prefetch (<link rel="prefetch">) is low-priority, idle-time. Preload (<link rel="preload">) is high-priority, in-parallel — use for chunks needed immediately after the current one.
prefetch.jsx
// Prefetch on hover — chunk in cache before user clicks
const handleHover = () => {
    import(/* webpackPrefetch: true */ './pages/Dashboard');
};

// Preload — needed immediately after current chunk (e.g. checkout after cart)
const Checkout = lazy(() =>
    import(/* webpackPreload: true */ './pages/Checkout'));
06

Runtime Bottlenecks

6.1 Long Tasks and Main-Thread Yielding
main threadscheduler
▼
A Long Task is any main-thread task exceeding 50ms — it blocks user input, animations, and React's own scheduler. React's concurrent scheduler yields to the browser every 5ms when using createRoot, but synchronous user code inside renders does not automatically yield. For CPU-bound work (large file parsing, heavy data transforms), manually chunk and yield between chunks.
yielding.js
// Chunk processing with yield — avoids Long Tasks blocking input/paint
async function processInChunks(items) {
    const CHUNK = 100;
    const results = [];
    for (let i = 0; i < items.length; i += CHUNK) {
        results.push(...items.slice(i, i + CHUNK).map(expensiveTransform));
        await new Promise(r => setTimeout(r, 0)); // yield to browser
    }
    return results;
}

// Chrome 115+ / modern — higher priority than setTimeout(0)
for (const chunk of chunks) {
    processChunk(chunk);
    await scheduler.yield();
}
6.2 Web Workers with Comlink — Offload CPU to a Separate Thread
main thread
▼
Web Workers run on a separate OS thread with no DOM access. Ideal for: cryptographic operations, large data transformations, WASM execution, and binary parsing. Comlink wraps the postMessage API in a transparent async proxy, making cross-thread calls look like normal function calls.
web-worker.js
// worker.js — separate thread, no DOM access
import * as Comlink from 'comlink';
Comlink.expose({ parseCSV: (raw) => heavyParse(raw) });

// main.js — transparent async proxy
const worker = new Worker(new URL('./worker.js', import.meta.url));
const remote = Comlink.wrap(worker);

// Runs off main thread — no jank
const rows = await remote.parseCSV(fileContents);
setData(rows);

React Profiling Strategy

7.1 React DevTools Profiler — Reading Flame Charts Correctly
profiling
▼
The Profiler shows why each component rendered (prop change, hook change, context change, parent render) and its self-render time. Grey bars rendered but committed without DOM changes — often wasted renders worth eliminating.

The Ranked chart sorts by self render time — look for components taking >16ms per interaction. The Timeline (React 18+) shows concurrent rendering slices and interaction attribution, correlating Long Tasks with specific user events.
Systematic profiling workflow 1. Record a specific interaction (one keystroke, one click).
2. Ranked chart → identify the slowest committing component.
3. Click it → "Why did this render?" shows the exact prop/hook/context that changed.
4. Fix: stabilise the reference, memoize the value, or split the context.
5. Record again — verify the component no longer appears in the ranked list.
7.2 Core Web Vitals as Code-Level Signals
profilingLCP · INP · CLS
▼
Understanding the causal chain from code pattern to Core Web Vital metric is a senior-level expectation.
LCP — Largest Contentful Paint Target <2.5s. Caused by unoptimised hero images, render-blocking JS, large initial bundles. Fix: preload LCP image, SSR/SSG above-fold content, code-split.
INP — Interaction to Next Paint Target <200ms. Replaced FID in March 2024. Caused by Long Tasks blocking response to input. Fix: useTransition, chunk Long Tasks, Web Workers.
CLS — Cumulative Layout Shift Target <0.1. Caused by images without dimensions, dynamically injected content above fold, font FOUT. Fix: CSS aspect-ratio, skeleton screens, font-display: swap.
web-vitals.js
import { onLCP, onINP, onCLS } from 'web-vitals'; // v3+

// INP replaced FID in March 2024 — update metric collection accordingly
onINP(({ name, value, rating, attribution }) => {
    analytics.track('web-vital', {
        name, value, rating,
        element:            attribution.interactionTarget,
        longAnimationFrames: attribution.longAnimationFrameEntries,
    });
});

React Senior Performance Checklist

— Architecture & Code Review Level
senior
▼
  • Understand the Fiber render vs. commit phase split. Never put side effects in render — the render phase can be run multiple times and thrown away.
  • Use useTransition for state updates that trigger expensive renders — updates become interruptible, not just deferred.
  • Use useDeferredValue + React.memo to defer expensive subtrees when you don't own the setter.
  • Know that React 18 automatic batching is a breaking change — use flushSync for DOM measurement between state updates.
  • Use stable, data-derived keys. Never use array index on sortable or filterable lists.
  • Intentionally change a key to force component remount and state reset — cleaner than multi-effect reset patterns.
  • Audit inline object/array literals in JSX and context provider values — these are the most common cause of memo-defeating re-renders.
  • Split contexts by update frequency, not domain. Use Zustand/Jotai for high-frequency shared state consumed at many tree depths.
  • Separate state context from dispatch context — dispatch identity is stable and never causes consumer re-renders.
  • Use React Query's select option to subscribe to derived slices — component re-renders only when the derived value changes.
  • Implement optimistic updates with rollback: onMutate sets cache, onError reverts, onSettled invalidates.
  • Declare React and ReactDOM as shared singletons in Module Federation — prevents duplicate instances and broken context across app boundaries.
  • Use webpackPrefetch to preload chunks during idle time and eliminate lazy-load waterfalls.
  • Identify Long Tasks (>50ms) in the Performance panel — break them with scheduler.yield() or chunk + setTimeout(0).
  • Offload CPU-intensive work (CSV parsing, crypto, WASM) to Web Workers via Comlink.
  • INP replaced FID as a Core Web Vital in March 2024 — update web-vitals library to v3+ and switch tracking from onFID to onINP.
  • Profile with React DevTools Profiler before adding useMemo/useCallback — verify the render is actually slow and that the memo comparison cost is lower than the render cost.
