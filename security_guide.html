Authentication & JWT

1.1 JWT — What the Spec Allows and What You Must Lock Down
authcritical
▼
JWT vulnerabilities arise almost entirely from trusting the token's own claims about how to validate it. The classic attacks:

Algorithm confusion (alg:none): Early JWT libraries accepted "alg":"none" in the header and skipped signature verification entirely. Always whitelist exactly the algorithms you use — reject everything else.

RS256 → HS256 downgrade: If a server using RSA (public key verifies) is tricked into using HMAC with the public key as the secret, the attacker who knows the public key can forge tokens. Explicitly specify the expected algorithm.

Missing aud / iss validation: A token issued by your auth server for Service A is also valid for Service B if B doesn't validate aud. Always validate both.
jwt-validation.cs
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(opts =>
    {
        opts.TokenValidationParameters = new TokenValidationParameters
        {
            // Whitelist — never trust alg from token header
            ValidAlgorithms       = [SecurityAlgorithms.RsaSha256],
            ValidateIssuerSigningKey = true,
            IssuerSigningKey      = rsaPublicKey,

            // Always validate issuer AND audience
            ValidateIssuer   = true,
            ValidIssuer      = "https://auth.example.com",
            ValidateAudience = true,
            ValidAudience    = "api.example.com",

            // Clock skew: 0 in strict environments; default is 5 min (too loose)
            ClockSkew        = TimeSpan.Zero,

            // Short-lived tokens — force refresh, limit blast radius on leak
            ValidateLifetime = true,
        };

        // Reject tokens with no expiry claim
        opts.TokenValidationParameters.RequireExpirationTime = true;
    });
Common production mistake Setting ClockSkew = TimeSpan.FromMinutes(5) (the default) means a stolen token that should have expired 4 minutes ago is still accepted. Set it to TimeSpan.Zero for APIs. Only allow skew when cross-region clock drift is a confirmed operational issue.
1.2 Refresh Token Rotation and Revocation
authcritical
▼
Access tokens should be short-lived (5–15 minutes). Refresh tokens are longer-lived and must be revocable — JWTs are stateless and cannot be revoked without a server-side store. Use opaque refresh tokens stored in your database with a family-tracking scheme.

Refresh token rotation: each use of a refresh token issues a new one and invalidates the old one. Refresh token reuse detection: if an already-used token is presented again, it indicates theft — invalidate the entire family immediately and force re-authentication.
refresh-token.cs
public async Task<TokenPair> RotateAsync(string refreshToken, CancellationToken ct)
{
    var stored = await _db.RefreshTokens
        .Include(r => r.Family)
        .FirstOrDefaultAsync(r => r.TokenHash == Hash(refreshToken), ct);

    if (stored is null)
        throw new SecurityException("Token not found");

    if (stored.IsRevoked)
    {
        // Reuse detected — invalidate ENTIRE family (theft indicator)
        await _db.RefreshTokens
            .Where(r => r.FamilyId == stored.FamilyId)
            .ExecuteUpdateAsync(r => r.SetProperty(x => x.IsRevoked, true), ct);
        throw new SecurityException("Token reuse detected — family revoked");
    }

    // Revoke used token, issue new one in same family
    stored.IsRevoked = true;
    var newRefresh = IssueRefreshToken(stored.FamilyId);
    var newAccess  = IssueAccessToken(stored.UserId);
    _db.RefreshTokens.Add(newRefresh);
    await _db.SaveChangesAsync(ct);

    return new TokenPair(newAccess, newRefresh.Token);
}
1.3 Password Hashing — PBKDF2, BCrypt, and Argon2
cryptocritical
▼
Never store plaintext or fast-hashed (MD5, SHA-256) passwords. Fast hash algorithms are designed for throughput — an attacker with a GPU can try billions of hashes per second. Password hashing must be computationally expensive and tunable.

ASP.NET Core Identity defaults to PBKDF2-HMAC-SHA512 with 600,000 iterations (v3 hasher) — acceptable baseline. Argon2id (via the Konscious.Security.Cryptography package) is the 2015 Password Hashing Competition winner and preferred where possible — it's memory-hard, resisting GPU and ASIC parallelism.
password-hashing.cs
// ASP.NET Core Identity — configure to use Argon2id via custom hasher
services.AddIdentity<AppUser, IdentityRole>()
    .AddDefaultTokenProviders();
services.AddScoped<IPasswordHasher<AppUser>, Argon2PasswordHasher<AppUser>>();

// Argon2id implementation — tune parameters for ~300ms on target hardware
public class Argon2PasswordHasher<T> : IPasswordHasher<T> where T : class
{
    public string HashPassword(T user, string password)
    {
        var salt = RandomNumberGenerator.GetBytes(16);
        using var hasher = new Argon2id(Encoding.UTF8.GetBytes(password))
        {
            Salt            = salt,
            DegreeOfParallelism = 1,
            MemorySize      = 65536,   // 64MB — resists GPU parallelism
            Iterations      = 3,
        };
        return Convert.ToBase64String(salt.Concat(hasher.GetBytes(32)).ToArray());
    }
}
Iteration count drift Tune hashing parameters so verification takes ~200–300ms on your production hardware. Re-tune every major hardware generation. ASP.NET Core Identity supports automatic re-hashing on login when a user's hash was created with older parameters — use PasswordHasherCompatibilityMode.

Authorization & Access Control

2.1 Policy-Based Authorization — Beyond [Authorize(Roles="Admin")]
authzOWASP A01
▼
Role-based checks hardcoded in attributes are brittle — roles proliferate, business rules get duplicated across controllers, and changes require redeployment. ASP.NET Core's policy-based authorization externalises rules into composable, testable handlers.

Resource-based authorization is the critical missing layer: a user may have the Editor role but should only be able to edit their own documents. This must be enforced in-handler, not at the routing layer, because the resource identity is only known after the request is parsed.
authorization.cs
// Register policy — encapsulates business rule, independently testable
services.AddAuthorization(opts =>
{
    opts.AddPolicy("SeniorEditor", policy => policy
        .RequireClaim("department", "editorial")
        .RequireClaim("seniority_years")
        .AddRequirements(new MinSeniorityRequirement(3)));
});

// Resource-based check — enforce INSIDE the handler, not at routing
[HttpPut("{id}")]
public async Task<IActionResult> UpdateDocument(int id, DocumentDto dto)
{
    var doc = await _docs.FindAsync(id);
    if (doc is null) return NotFound();

    // Resource-based: check ownership AFTER loading the resource
    var auth = await _authorizationService
        .AuthorizeAsync(User, doc, "DocumentEditPolicy");
    if (!auth.Succeeded) return Forbid();

    // proceed with update
}
2.2 IDOR — Insecure Direct Object Reference
criticalOWASP A01
▼
IDOR is the most common access control vulnerability: an authenticated user increments an ID in the URL and accesses another user's data. The fix is always the same — scope every data query to the current user's tenant or identity. Never trust the ID from the request to prove ownership.
❌ IDOR — trusts request ID GET /api/orders/7891

var order = await _db.Orders
.FindAsync(id); // any user can access any order
✅ Scoped to authenticated user var order = await _db.Orders
.Where(o => o.Id == id
&& o.UserId == currentUserId)
.FirstOrDefaultAsync(ct);
Multi-tenant systems In multi-tenant apps, always include the TenantId in every query. Consider a global query filter on the DbContext that automatically appends the tenant scope — prevents accidental cross-tenant data leaks from missing WHERE clauses.
global-tenant-filter.cs
// DbContext — global query filter automatically scopes all queries
protected override void OnModelCreating(ModelBuilder builder)
{
    builder.Entity<Order>()
        .HasQueryFilter(o => o.TenantId == _tenantId);
    builder.Entity<Customer>()
        .HasQueryFilter(c => c.TenantId == _tenantId);
    // All queries automatically scoped — must use .IgnoreQueryFilters() to bypass
}
2.3 Mass Assignment — Over-Posting and Model Binding
inputOWASP A08
▼
Mass assignment (over-posting) occurs when model binding maps request fields directly onto a domain entity, allowing attackers to set fields the API never intended to expose — e.g. IsAdmin=true, Balance=99999. Always bind to a dedicated DTO that only exposes the fields the API accepts, then map to the domain model.
mass-assignment.cs
// ❌ Binds directly to entity — attacker can POST IsAdmin=true
[HttpPost] public IActionResult Create([FromBody] User user) { ... }

// ✅ Bind to DTO — only fields the API explicitly accepts
public record CreateUserDto(
    [Required, MaxLength(100)] string Name,
    [Required, EmailAddress]    string Email,
    [Required, MinLength(8)]   string Password
);

[HttpPost]
public async Task<IActionResult> Create([FromBody] CreateUserDto dto)
{
    var user = new User
    {
        Name     = dto.Name,
        Email    = dto.Email.ToLowerInvariant(),
        IsAdmin  = false,  // ← always set server-side, never from input
        TenantId = _tenantId // ← always set from authenticated context
    };
    user.PasswordHash = _hasher.HashPassword(user, dto.Password);
}

Injection & Input Validation

3.1 SQL Injection — Parameterisation and ORM Pitfalls
criticalOWASP A03injection
▼
SQL injection is fully prevented by parameterised queries. EF Core always parameterises LINQ queries. The vulnerabilities arise in two specific cases: raw SQL methods (FromSqlRaw, ExecuteSqlRaw) with string interpolation, and dynamic ORDER BY / column names which cannot be parameterised by the database driver and must be whitelisted manually.
sql-injection.cs
// ❌ String interpolation in raw SQL — injectable
var result = _db.Users.FromSqlRaw($"SELECT * FROM Users WHERE Name = '{name}'");

// ✅ Use FormattableString overload — EF Core parameterises it
var result = _db.Users.FromSql($"SELECT * FROM Users WHERE Name = {name}");

// ✅ Explicit SqlParameter
var result = _db.Users.FromSqlRaw(
    "SELECT * FROM Users WHERE Name = @name",
    new SqlParameter("@name", name));

// ✅ Dynamic ORDER BY — whitelist column names, never interpolate raw input
private static readonly HashSet<string> _allowedSortCols =
    ["Name", "CreatedAt", "Status"];

if (!_allowedSortCols.Contains(sortBy))
    throw new ArgumentException("Invalid sort column");

var result = _db.Orders.FromSqlRaw(
    $"SELECT * FROM Orders ORDER BY [{sortBy}] {(descending ? "DESC" : "ASC")}");
3.2 Input Validation — FluentValidation and Defence in Depth
validationOWASP A03
▼
Data annotations ([Required], [MaxLength]) are a first-pass UI hint, not a security boundary — they can be bypassed by sending a raw HTTP request. FluentValidation enforces rules at the application layer, after model binding. Combine with server-side validation that is always exercised regardless of client state.

Critical: validate file uploads by magic bytes (not MIME type from the Content-Type header, which is attacker-controlled), enforce size limits, and never derive storage paths from user-supplied filenames.
validation.cs
public class CreateOrderValidator : AbstractValidator<CreateOrderDto>
{
    public CreateOrderValidator()
    {
        RuleFor(x => x.CustomerId)
            .NotEmpty()
            .MustAsync(async (id, ct) => await _customerService.ExistsAsync(id, ct))
            .WithMessage("Customer not found");

        RuleFor(x => x.Items)
            .NotEmpty()
            .ForEach(item =>
            {
                item.ChildRules(i =>
                {
                    i.RuleFor(x => x.Quantity).InclusiveBetween(1, 100);
                    i.RuleFor(x => x.UnitPrice).GreaterThan(0);
                });
            });

        // File upload — validate by magic bytes, not Content-Type
        RuleFor(x => x.Attachment)
            .Must(file => IsValidPdf(file))
            .WithMessage("Only PDF attachments accepted");
    }

    private static bool IsValidPdf(IFormFile file)
    {
        var magic = new byte[4];
        file.OpenReadStream().Read(magic);
        return magic.SequenceEqual(new byte[] { 0x25, 0x50, 0x44, 0x46 }); // %PDF
    }
}
3.3 Path Traversal and File Upload Security
injectioncritical
▼
Path traversal (../../../etc/passwd) occurs when user input is used to construct file system paths. Never construct a path from user input directly. Generate a server-side UUID filename, store the original name as metadata only.
file-upload.cs
// ❌ Path traversal — attacker sends filename = "../../appsettings.json"
var path = Path.Combine(_uploadDir, file.FileName);

// ✅ Generate server-side name — user-supplied name stored as metadata only
var storedName = $"{Guid.NewGuid():N}{Path.GetExtension(file.FileName)}";
var fullPath   = Path.GetFullPath(Path.Combine(_uploadDir, storedName));

// Canonicalise and ensure the path stays within the upload directory
if (!fullPath.StartsWith(Path.GetFullPath(_uploadDir), StringComparison.OrdinalIgnoreCase))
    throw new SecurityException("Path traversal detected");

await using var stream = File.Create(fullPath);
await file.CopyToAsync(stream, ct);

Cryptography

4.1 Encryption — AES-GCM vs AES-CBC and When to Use Each
crypto
▼
AES-CBC provides confidentiality only — it does not authenticate the ciphertext. An attacker can flip bits in the ciphertext to predictably flip bits in the plaintext (padding oracle attacks, CBC bit-flipping). Never use CBC without a separate MAC (Encrypt-then-MAC).

AES-GCM (Galois/Counter Mode) provides authenticated encryption — confidentiality + integrity in a single primitive. The authentication tag detects any tampering. Use AES-GCM for all new symmetric encryption. Critical: the nonce (IV) must be unique per encryption operation — reusing a nonce with the same key destroys both confidentiality and the authentication tag.
aes-gcm.cs
public static (byte[] ciphertext, byte[] nonce, byte[] tag) Encrypt(
    ReadOnlySpan<byte> plaintext, byte[] key)
{
    var nonce      = new byte[AesGcm.NonceByteSizes.MaxSize]; // 12 bytes
    var ciphertext = new byte[plaintext.Length];
    var tag        = new byte[AesGcm.TagByteSizes.MaxSize];   // 16 bytes

    RandomNumberGenerator.Fill(nonce); // cryptographically random nonce — never reuse

    using var aes = new AesGcm(key, tag.Length);
    aes.Encrypt(nonce, plaintext, ciphertext, tag);

    return (ciphertext, nonce, tag);
}

public static byte[] Decrypt(
    ReadOnlySpan<byte> ciphertext, byte[] key, byte[] nonce, byte[] tag)
{
    var plaintext = new byte[ciphertext.Length];
    using var aes = new AesGcm(key, tag.Length);
    aes.Decrypt(nonce, ciphertext, tag, plaintext);
    // Throws AuthenticationTagMismatchException if ciphertext was tampered
    return plaintext;
}
4.2 Data Protection API — Secure Token and Cookie Encryption
cryptoauth
▼
ASP.NET Core's Data Protection API handles key management, rotation, and authenticated encryption for application-layer tokens (anti-forgery tokens, password reset links, cookie values). Use it instead of rolling your own AES wrapper for app-level secrets. In distributed deployments, configure a shared key ring via Azure Key Vault, Redis, or a database — otherwise tokens encrypted on one instance cannot be decrypted on another.
data-protection.cs
// Configure shared key ring for distributed deployments
services.AddDataProtection()
    .PersistKeysToAzureBlobStorage(blobClient)
    .ProtectKeysWithAzureKeyVault(keyVaultKeyId, new DefaultAzureCredential())
    .SetApplicationName("MyApp")                // isolates from other apps
    .SetDefaultKeyLifetime(TimeSpan.FromDays(90)); // rotate keys every 90 days

// Use for password reset tokens — purpose-scoped, tamper-evident
var protector = _dataProtection.CreateProtector("PasswordReset.v1");
var token     = protector.Protect($"{userId}:{expiry:O}");

// Verify — decryption fails if tampered or wrong purpose
var payload = protector.Unprotect(token); // throws CryptographicException if invalid

Security Headers & CORS

5.1 CORS Misconfiguration — The Wildcard and Origin Reflection Traps
CORScritical
▼
Two CORS misconfigurations are especially dangerous:

Wildcard + credentials: Access-Control-Allow-Origin: * cannot be combined with Access-Control-Allow-Credentials: true — browsers reject it. Developers sometimes work around this by reflecting the Origin header dynamically (Allow-Origin: <whatever the request said>), effectively allowing any origin to make authenticated cross-origin requests. Attackers exploit this from any domain.

Null origin: some implementations allow Origin: null — sent by sandboxed iframes, file://, and data URIs. An attacker using a sandboxed iframe can trigger cross-origin requests from null origin.
cors.cs
builder.Services.AddCors(opts =>
{
    opts.AddPolicy("Frontend", policy => policy
        // ✅ Explicit allowlist — never reflect Origin dynamically
        .WithOrigins("https://app.example.com", "https://www.example.com")
        .WithMethods("GET", "POST", "PUT", "DELETE")
        .WithHeaders("Authorization", "Content-Type")
        .AllowCredentials()   // safe only with explicit origins above
        .SetPreflightMaxAge(TimeSpan.FromMinutes(10)));
});

// ❌ Never do this — reflects arbitrary Origin
// .AllowAnyOrigin().AllowCredentials()  ← not possible; browsers block it
// Workaround people use (dangerous):
// context.Response.Headers["Access-Control-Allow-Origin"] = context.Request.Headers["Origin"];
5.2 Security Response Headers — HSTS, CSP, and Friends
headersbrowser
▼
Security headers are defence-in-depth browser controls — they limit the blast radius of other vulnerabilities. Senior engineers know which header does what and why.
HSTS Strict-Transport-Security — forces HTTPS for max-age seconds. Include includeSubDomains and submit to the HSTS preload list for new domains.
X-Content-Type-Options nosniff — prevents browsers from MIME-sniffing a response away from the declared Content-Type. Stops script execution from uploaded files served as text/plain.
X-Frame-Options / CSP frame-ancestors Clickjacking prevention. DENY or SAMEORIGIN. Prefer CSP frame-ancestors — X-Frame-Options is superseded and less granular.
security-headers.cs
app.Use(async (context, next) =>
{
    var headers = context.Response.Headers;
    headers.Append("Strict-Transport-Security",
        "max-age=63072000; includeSubDomains; preload");
    headers.Append("X-Content-Type-Options",    "nosniff");
    headers.Append("X-Frame-Options",            "DENY");
    headers.Append("Referrer-Policy",
        "strict-origin-when-cross-origin");
    headers.Append("Permissions-Policy",
        "camera=(), microphone=(), geolocation=()");
    // CSP set separately per environment — see React section for detail
    await next(context);
});
5.3 Rate Limiting and Anti-Enumeration
networkauth
▼
Rate limiting belongs at the application layer, not just the gateway — a misconfigured gateway or a direct-access path bypasses gateway limits entirely. ASP.NET Core 7+ includes built-in rate limiting middleware. Critical patterns:

Credential stuffing prevention: rate-limit login by IP and by username independently — an attacker rotating IPs but targeting the same account should still be blocked. Enumeration prevention: login, forgot-password, and account-existence endpoints must return identical responses and take identical time for found vs. not-found users — otherwise attackers enumerate valid accounts.
rate-limiting.cs
services.AddRateLimiter(opts =>
{
    // Sliding window on login — per IP
    opts.AddSlidingWindowLimiter("login", limiterOpts =>
    {
        limiterOpts.PermitLimit         = 5;
        limiterOpts.Window              = TimeSpan.FromMinutes(15);
        limiterOpts.SegmentsPerWindow   = 3;
        limiterOpts.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
        limiterOpts.QueueLimit          = 0;
    });

    opts.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});

// Anti-enumeration — identical timing regardless of user existence
public async Task<IActionResult> Login(LoginDto dto)
{
    var user = await _users.FindByEmailAsync(dto.Email);

    // Always verify hash — even if user not found (prevent timing oracle)
    var dummyHash = "$argon2id$..."; // pre-computed dummy
    var hashToCheck = user?.PasswordHash ?? dummyHash;
    var valid = _hasher.VerifyPassword(hashToCheck, dto.Password);

    if (user is null || !valid) return Unauthorized(); // same response
    return Ok(await IssueTokensAsync(user));
}

Secrets Management

6.1 Never in Code, Never in Config Files — Managed Identity First
criticalOWASP A02
▼
Secrets in appsettings.json, environment variables in Dockerfiles, or connection strings in config files are routinely leaked via Git history, log files, error pages, and container image layers. The architecture goal is zero secrets in application code or config.

Managed Identity (Azure) / IAM roles (AWS) / Workload Identity (GCP) allow the compute resource itself to authenticate to other services without any credential. The platform manages token issuance and rotation. Use this for: database connections, storage accounts, message queues, and vault access.
managed-identity.cs
// Azure — no connection string, no password; Managed Identity authenticates
services.AddDbContext<AppDbContext>(opts =>
    opts.UseSqlServer(builder.Configuration.GetConnectionString("Db"),
        sql => sql.UseAzureActiveDirectoryAuthentication(
            new DefaultAzureCredential()))); // no password in config

// Azure Key Vault — load secrets at startup via Managed Identity
builder.Configuration.AddAzureKeyVault(
    new Uri($"https://{vaultName}.vault.azure.net"),
    new DefaultAzureCredential());
// Secrets now available via IConfiguration["MySecret"]
// Rotation: Key Vault reloads secret on each Configuration.Reload()
Secret rotation Secrets that cannot be rotated without downtime are a liability. Design connection strings and API keys to support dual-active rotation: load both current and next secret, try both on failure, retire old after rotation completes. Key Vault versioning supports this natively.

Secure Logging & Audit

7.1 Log Injection, PII Leakage, and What Not to Log
injectionOWASP A09
▼
Log injection: if user-supplied strings containing newlines are logged unsanitised, attackers can forge log entries — fabricating events, hiding real ones, or injecting malicious data into log aggregation systems. Always sanitise or structure-log untrusted data.

PII and secret leakage in logs is a compliance and security incident waiting to happen. Common culprits: logging the full request body (which may contain passwords or payment data), exception messages that include query strings with tokens, and structured log serialization that deep-serializes objects containing credential fields.
secure-logging.cs
// ❌ Logs raw user input — injectable, may contain PII
_logger.LogInformation("User searched for: {query}", rawUserInput);

// ✅ Sanitise: strip control characters before logging untrusted data
var safe = Regex.Replace(rawUserInput, @"[\r\n\t]", " ")[..Math.Min(rawUserInput.Length, 200)];
_logger.LogInformation("User searched for: {query}", safe);

// ✅ Structured logging with explicit field inclusion — no object deep-serialize
_logger.LogInformation("Order placed {OrderId} by {UserId} for {TotalAmount}",
    order.Id, order.UserId, order.Total);
// Never: _logger.LogInformation("Order {order}", order); — may log card data

// ✅ Audit log — separate, tamper-evident, append-only store
await _auditLog.WriteAsync(new AuditEntry
{
    Action    = "OrderCancelled",
    UserId    = User.GetUserId(),
    IpAddress = context.Connection.RemoteIpAddress?.ToString(),
    Timestamp = DateTimeOffset.UtcNow,
    EntityId  = orderId.ToString(),
    // No sensitive field values in audit log — just the action and identity
});
Never log Passwords, API keys, JWT tokens, credit card numbers, SSNs, full request bodies for auth endpoints, session cookies, or any field marked as sensitive in your data classification schema.

.NET Security Checklist

— Application & Architecture Level
senior
▼
  • Whitelist JWT algorithms explicitly — never trust alg from the token header. Reject none and unexpected algorithms.
  • Validate iss, aud, exp on every JWT. Set ClockSkew = TimeSpan.Zero for strict environments.
  • Implement refresh token rotation with reuse detection — on reuse, invalidate the entire token family.
  • Scope every data query to the authenticated user/tenant — never trust a resource ID from the request to prove ownership (IDOR).
  • Use DTOs for model binding — never bind request data directly onto domain entities (mass assignment / over-posting).
  • Use FromSql($"") (interpolated overload) or explicit SqlParameter — never string-interpolate into FromSqlRaw.
  • Hash passwords with Argon2id or PBKDF2 ≥600k iterations. Tune to ~200–300ms on production hardware. Re-hash on login when parameters are outdated.
  • Use AES-GCM for symmetric encryption — never AES-CBC without a separate MAC. Generate a fresh random nonce per encryption.
  • Use ASP.NET Core Data Protection API for app-layer tokens. Configure shared key ring for distributed deployments.
  • Configure CORS with an explicit origin allowlist. Never dynamically reflect the Origin header.
  • Set HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy on all responses.
  • Apply rate limiting at the application layer (not only gateway) — limit by IP and by username independently on auth endpoints.
  • Make login/forgot-password endpoints timing-constant — always hash even for non-existent users to prevent enumeration.
  • Use Managed Identity / IAM roles for service-to-service auth — zero secrets in config or environment variables.
  • Load secrets from a vault (Key Vault, AWS Secrets Manager) — design for rotation without downtime via dual-active secret loading.
  • Validate file uploads by magic bytes, not Content-Type header. Generate server-side filenames — never use user-supplied names for storage paths.
  • Sanitise user input in log messages (strip \r\n\t). Never log passwords, tokens, card numbers, or full request bodies of auth endpoints.
  • Apply global EF Core query filters per tenant. Audit any use of .IgnoreQueryFilters() — it bypasses all tenant/soft-delete scoping.
  • Review exception handling: never return stack traces, SQL error messages, or internal paths in API error responses.
Part 2 — React Application Security
01

XSS & DOM Injection

1.1 React's XSS Model — What It Prevents and What It Doesn't
criticalOWASP A03browser
▼
React automatically HTML-encodes all values interpolated in JSX — {userInput} renders as text, not HTML, so a string like <script>alert(1)</script> is displayed literally. This prevents reflected and stored XSS through JSX rendering.

What React does not protect against:

1. dangerouslySetInnerHTML — bypasses React's encoding entirely. Raw HTML is injected into the DOM. Only use with output from a trusted server-side HTML sanitizer.

2. href with javascript: URLs — React does not sanitize URL-scheme attributes. An attacker-controlled URL of javascript:alert(document.cookie) executes JavaScript when clicked.

3. DOM-based XSS via eval, innerHTML, or template literals — imperative DOM manipulation outside React's rendering pipeline is entirely unprotected.
xss.jsx
// ✅ Safe — React HTML-encodes this automatically
<p>{userInput}</p>

// ❌ Dangerous — bypasses encoding; attacker-controlled HTML injected
<div dangerouslySetInnerHTML={{ __html: userContent }} />

// ✅ Only use with DOMPurify output from a trusted sanitiser
import DOMPurify from 'dompurify';
const clean = DOMPurify.sanitize(userContent, {
    ALLOWED_TAGS:  ['b', 'i', 'em', 'strong', 'a', 'p'],
    ALLOWED_ATTR:  ['href', 'title'],
    ALLOW_DATA_ATTR: false,
});
<div dangerouslySetInnerHTML={{ __html: clean }} />

// ❌ javascript: URL — executes on click
<a href={userUrl}>Click</a>

// ✅ Validate URL scheme before rendering
function SafeLink({ url, children }) {
    const safe = url.startsWith('https://') || url.startsWith('http://');
    return safe
        ? <a href={url} rel="noopener noreferrer">{children}</a>
        : <span>{children}</span>;
}
1.2 Server-Side Rendering XSS — State Hydration Injection
criticalSSR
▼
SSR introduces a specific XSS vector: when server-side Redux/React Query state is serialized into a <script> tag for hydration, user-controlled strings containing </script> break out of the script context.
❌ Script injection via JSON // Server renders:
<script>

CSRF & Request Forgery

2.1 CSRF — When It Applies and the SameSite Cookie Nuance
OWASP A01CSRF
▼
CSRF exploits the browser's automatic cookie attachment on cross-origin requests. It only applies when your API authenticates via cookies. APIs that authenticate via Authorization header (Bearer tokens stored in memory, not cookies) are not vulnerable to CSRF — cross-origin requests cannot read or set the Authorization header.

SameSite=Strict cookies are never sent on cross-origin requests — completely prevents CSRF but breaks legitimate cross-origin flows (OAuth redirects, links from emails). SameSite=Lax allows top-level navigations (GET) but blocks cross-origin POSTs — good default. SameSite=None; Secure must be combined with an explicit CSRF token.
csrf.tsx
// ✅ If using cookie auth: read CSRF token from a non-HttpOnly cookie
// and attach it as a custom header — browsers block cross-origin custom headers
const getCsrfToken = () =>
    document.cookie
        .split(';')
        .find(c => c.trim().startsWith('csrf='))
        ?.split('=')[1];

const apiClient = axios.create({ withCredentials: true });
apiClient.interceptors.request.use(config => {
    if (['post', 'put', 'patch', 'delete'].includes(config.method!)) {
        config.headers['X-CSRF-Token'] = getCsrfToken() ?? '';
    }
    return config;
});

// ✅ Preferred: Bearer tokens in memory — zero CSRF exposure
// httpClient.defaults.headers.common['Authorization'] = `Bearer ${accessToken}`
// No cookie → no CSRF → no CSRF token needed

Client Storage Security

3.1 Where to Store Tokens — The XSS vs CSRF Tradeoff
criticalstorage
▼
There is no perfectly safe place to store tokens in the browser — only different threat profiles. Senior engineers must articulate the tradeoff explicitly.
localStorage / sessionStorage XSS risk. Any injected script reads tokens with localStorage.getItem(). Not accessible to HttpOnly cookies. Use only for low-sensitivity, short-lived tokens in environments with strict CSP.
HttpOnly Cookie XSS-resistant, CSRF risk. Script cannot read it. Requires CSRF token mitigation if not using SameSite=Strict. Best for access tokens when SSR is involved.
In-memory (module variable / closure) Best for access tokens. Not persistent across page refresh — use a refresh token in HttpOnly cookie to rehydrate. No XSS access, no CSRF surface. Clears on tab close.
token-storage.ts
// In-memory token store — XSS cannot access, CSRF not applicable let accessToken: string | null = null; export const tokenStore = { set: (t: string) => { accessToken = t; }, get: () => accessToken, clear: () => { accessToken = null; }, }; // On app init: silently refresh using HttpOnly cookie holding refresh token async function initAuth() { try { const { accessToken } = await api.post('/auth/refresh'); // withCredentials: true — browser sends HttpOnly refresh cookie automatically tokenStore.set(accessToken); } catch { /* unauthenticated — prompt login */ } }

Content Security Policy

4.1 CSP — Nonce-Based vs Hash-Based vs unsafe-inline
CSPOWASP A05
▼
CSP is the last line of defence against XSS — it restricts what the browser will execute even if an attacker injects a script tag. The challenge for SPAs is that React typically requires inline scripts for hydration and dynamic imports.

unsafe-inline defeats the entire purpose of CSP — allows any inline script. Nonce-based CSP generates a server-side random nonce per request, adds it to the header, and only allows script tags bearing that nonce — dynamic, but requires server-side rendering or a middleware layer. Hash-based CSP precomputes SHA-256 hashes of known inline scripts — works for static inline scripts but breaks if the content changes.
csp.ts
// Next.js middleware — nonce-based CSP per request
import { NextResponse } from 'next/server';
import { randomBytes } from 'crypto';

export function middleware(request: Request) {
    const nonce = randomBytes(16).toString('base64');

    const csp = [
        `default-src 'self'`,
        `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'`,
        // strict-dynamic: trusted nonce'd scripts can load other scripts;
        // all other scripts blocked — even from allowlisted hosts
        `style-src 'self' 'nonce-${nonce}'`,
        `img-src 'self' https://cdn.example.com data:`,
        `font-src 'self'`,
        `connect-src 'self' https://api.example.com`,
        `frame-ancestors 'none'`,
        `form-action 'self'`,
        `base-uri 'self'`,
        `report-uri /api/csp-report`,    // collect violations
    ].join(';');

    const response = NextResponse.next();
    response.headers.set('Content-Security-Policy', csp);
    response.headers.set('x-nonce', nonce); // pass to _document for script tags
    return response;
}
Deploy in report-only first Use Content-Security-Policy-Report-Only header before enforcing. Collect violations for 1–2 weeks via report-uri to discover legitimate scripts that need nonces before flipping to enforcement mode.

Dependency & Supply Chain Security

5.1 NPM Supply Chain — Typosquatting, Lifecycle Scripts, and Lockfiles
supply chainOWASP A06
▼
The NPM ecosystem has seen significant supply-chain attacks: typosquatting (malicious packages named similarly to popular ones), dependency confusion (publishing a private package name publicly at a higher version), and package takeover (maintainer account compromise).

Lifecycle scripts (postinstall, preinstall) run automatically on npm install — malicious packages use them to exfiltrate environment variables, tokens, and SSH keys at install time on developer machines and CI pipelines.

Auth Token Handling in React

6.1 Silent Refresh and Token Leakage via postMessage
authbrowser
▼
The silent refresh pattern (hidden iframe or popup window calling the auth server to get a new access token) carries a postMessage XSS risk if the message origin is not validated. A malicious script on any page listening to window.message events can receive the access token if the iframe's postMessage call targets "*" as the origin.
silent-refresh.ts
// ❌ Sends token to any origin — any page listening can steal it
window.parent.postMessage({ token: accessToken }, '*');

// ✅ Always specify exact target origin
window.parent.postMessage(
    { token: accessToken },
    'https://app.example.com'   // ← exact origin
);

// ✅ Receiver always validates origin before using message data
window.addEventListener('message', (event) => {
    if (event.origin !== 'https://auth.example.com') return; // discard
    const { token } = event.data;
    tokenStore.set(token);
});
6.2 Leaking Tokens in URLs and Browser History
authbrowser
▼
OAuth 2.0 implicit flow returned tokens in the URL fragment — those tokens are logged in browser history, server access logs for single-page navigations, and the Referer header on subsequent requests. Always use Authorization Code + PKCE for SPAs. The authorization code in the URL is short-lived and single-use; the actual tokens are exchanged server-side (or via the token endpoint) and never appear in the URL.
pkce.ts
// PKCE — code verifier generated in browser, code challenge sent to auth server
async function generatePkce() {
    const verifier  = randomBase64Url(32);
    const digest    = await crypto.subtle.digest(
        'SHA-256',
        new TextEncoder().encode(verifier)
    );
    const challenge = base64UrlEncode(new Uint8Array(digest));
    sessionStorage.setItem('pkce_verifier', verifier); // short-lived, session-only
    return { verifier, challenge };
}

// Authorization URL — code_challenge replaces implicit token in fragment
const { challenge } = await generatePkce();
const authUrl = `${AUTH_SERVER}/authorize?
    response_type=code
    &client_id=${CLIENT_ID}
    &redirect_uri=${REDIRECT_URI}
    &code_challenge=${challenge}
    &code_challenge_method=S256
    &scope=openid profile`.replace(/\s/g, '');

Secure Rendering Patterns

7.1 Prototype Pollution via JSON.parse and Object Spread
injectionruntime
▼
Prototype pollution occurs when attacker-controlled JSON is merged into objects without sanitisation, allowing modification of Object.prototype. A payload like {"__proto__": {"isAdmin": true}} merged with Object.assign({}, userInput) sets isAdmin on every object in the process.

Libraries like lodash.merge, deepmerge, and custom recursive mergers are vulnerable. Use JSON.parse with a reviver to reject dangerous keys, or use libraries with explicit prototype pollution protection (deepmerge v4.3+ with isMergeableObject, or structuredClone for deep copies).
proto-pollution.ts
// ❌ Vulnerable to prototype pollution if input is attacker-controlled
const merged = _.merge({}, userSuppliedConfig);

// ✅ Reviver rejects __proto__, constructor, prototype keys
const safe = JSON.parse(untrustedJson, (key, value) => {
    if (['__proto__', 'constructor', 'prototype'].includes(key)) {
        return undefined; // drop dangerous keys
    }
    return value;
});

// ✅ structuredClone — deep copy without prototype chain
const safeCopy = structuredClone(userInput);
7.2 Third-Party Scripts — Isolation and Integrity
supply chainbrowser
▼
Third-party scripts (analytics, chat widgets, A/B testing) run with full access to the page — they can read DOM content including form values, steal tokens from localStorage, and exfiltrate data to external servers. The 2019 British Airways breach (Magecart attack) was a compromised third-party script that exfiltrated payment card data from 500,000 customers.

Mitigations: (1) Subresource Integrity (SRI) — hash-pins CDN scripts, rejects tampered versions. (2) Sandboxed iframes — load third-party widgets in iframes with sandbox attribute, limiting their API access. (3) CSP connect-src — blocks data exfiltration by restricting outbound HTTP origins. (4) Minimise the number of third-party scripts; audit each one.
third-party.html
<!-- SRI — browser rejects if CDN script is tampered -->
<script
  src="https://cdn.example.com/analytics.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
  crossorigin="anonymous"
  defer></script>

<!-- Sandboxed iframe for third-party widget -->
<iframe
  src="https://widget.vendor.com/chat"
  sandbox="allow-scripts allow-same-origin allow-forms"
  <!-- allow-same-origin needed for localStorage; omit if not needed -->
  allow="camera 'none'; microphone 'none'">
</iframe>
7.3 Sensitive Data in React State and DevTools Exposure
runtime
▼
React DevTools (browser extension and bundled dev overlay) expose the full component tree and all state values to anyone with access to the browser — including malicious browser extensions. In development, this is acceptable. In production:

Never store sensitive values in React state (full card numbers, SSNs, decrypted secrets). Keep in-memory token store outside React's state graph (plain module variable, not useState/useContext). DevTools cannot inspect plain module-level variables.
sensitive-state.ts
// ❌ Token in React state — visible in DevTools
const [accessToken, setAccessToken] = useState<string | null>(null);

// ✅ Module-level variable — outside React's introspectable tree
let _accessToken: string | null = null;
export const tokenStore = {
    set: (t: string) => { _accessToken = t; },
    get: ()           => _accessToken,
    clear: ()         => { _accessToken = null; },
};

// Disable React DevTools in production builds
if (process.env.NODE_ENV === 'production') {
    const noop = () => {};
    (window as any).__REACT_DEVTOOLS_GLOBAL_HOOK__ = {
        supportsFiber: true,
        inject: noop, onCommitFiberRoot: noop, onCommitFiberUnmount: noop,
    };
}

React Security Checklist

— Application & Architecture Level
senior
▼
  • Audit every use of dangerouslySetInnerHTML — it must only receive output from a server-side sanitizer (DOMPurify with strict allowlist), never raw user input.
  • Validate URL schemes before rendering in href, src, or action — reject javascript:, data:, and vbscript: schemes.
  • In SSR, escape </script> in JSON hydration payloads — use serialize-javascript or equivalent, never raw JSON.stringify inside <script> tags.
  • Store access tokens in memory (module variable), not localStorage or React state. Rehydrate via HttpOnly refresh token cookie on page load.
  • CSRF only applies to cookie-authenticated APIs. If using Bearer tokens in Authorization header, CSRF is not a concern — don't build CSRF infrastructure unnecessarily.
  • Use Authorization Code + PKCE for OAuth flows in SPAs — never implicit flow. The authorization code appears in the URL; tokens must not.
  • Always specify exact target origin in postMessage calls. Always validate event.origin before acting on received messages.
  • Implement a strict nonce-based CSP. Use Content-Security-Policy-Report-Only in staging before enforcing. Never use unsafe-inline as a permanent solution.
  • Commit and verify package-lock.json. Run npm audit in CI with --audit-level=high. Use --ignore-scripts in production installs.
  • Use Subresource Integrity (SRI) hashes for all CDN-loaded scripts and stylesheets.
  • Sandbox third-party widgets in iframes with the sandbox attribute. Restrict connect-src in CSP to block data exfiltration.
  • Sanitize JSON from untrusted sources with a reviver that rejects __proto__, constructor, and prototype keys before merging into objects.
  • Keep sensitive values out of React state and Context — they are visible in React DevTools and browser extension introspection.
  • Disable React DevTools in production builds by replacing the global hook with no-ops.
  • Audit rel="noopener noreferrer" on all external links with target="_blank" — without it, the opened page can reference the opener via window.opener.
  • Review environment variables prefixed with REACT_APP_ / NEXT_PUBLIC_ — they are bundled into the client and publicly visible. Never put secrets in public env vars.
